# 사용자 컴포넌트 / 액션 가이드

이 문서는 `KirbyGameDll` 안에서 사용자 컴포넌트와 사용자 액션을 추가할 때,
어떤 흐름으로 작업하면 덜 헷갈리는지 정리한 문서입니다.

이 프로젝트에서는 엔진 코어를 직접 고치기보다,
`KirbyGameDll` 안에서 게임 로직을 확장하는 방식으로 작업하는 경우가 많습니다.

## 1. 어디에 추가하는가

사용자 컴포넌트는 보통 아래 폴더에 만듭니다.

```text
KirbyGameDll/UserComponents/Scripts
```

사용자 액션은 보통 아래 폴더에 만듭니다.

```text
KirbyGameDll/UserActions/Scripts
```

## 2. 사용자 컴포넌트 기본 흐름

새 컴포넌트를 만들 때는 보통 아래 순서로 진행합니다.

1. `MyComponent.h`, `MyComponent.cpp` 파일 생성
2. `Component` 상속
3. `kComponentType` 정의
4. `GameComponentRegistry.cpp` 등록
5. 필요하면 Inspector, 저장/로드, 참조 필드 구현

기본 형태 예시:

```cpp
class MyComponent : public Component
{
public:
    static constexpr const char* kComponentType = "MyComponent";

    void Initialize() override;
    void Release() override;
    void Start() override;
    void Update() override;

    const char* GetInspectorName() const override;
    const char* GetSerializableType() const override;
};
```

## 3. 이름은 서로 맞춰 둔다

아래 값은 서로 맞춰 두는 편이 좋습니다.

- `kComponentType`
- `GetSerializableType()` 반환값
- `GameComponentRegistry.cpp`에 등록하는 타입 이름

이 값이 서로 다르면 SceneData 저장/로드나 Inspector 연결이 꼬일 수 있습니다.

## 4. 컴포넌트 등록 위치

새 컴포넌트는 아래 파일에서 등록합니다.

```text
KirbyGameDll/UserComponents/GameComponentRegistry.cpp
```

예시:

```cpp
REGISTER_GAME_COMPONENT(factory, MyComponent, "My Component", "User");
```

여기서 보통 보는 값은 아래 정도입니다.

- 실제 C++ 타입
- 에디터에 보일 이름
- 카테고리 이름

## 5. Inspector 만들기

에디터에서 값을 조절하고 싶으면 `DrawInspector()`를 구현합니다.

예시:

```cpp
void MyComponent::DrawInspector()
{
    SyncImGuiContextForCurrentModule();
    ImGui::Checkbox("Enabled", &m_enabled);
    ImGui::DragFloat("Speed", &m_speed, 0.1f);
}
```

주의할 점:

- `KirbyGameDll` 쪽 Inspector에서는 보통
  `SyncImGuiContextForCurrentModule();`
  를 먼저 호출하는 편이 안전합니다.

## 6. 참조 필드가 있으면

`GameObject*`, `Component*` 같은 참조를 Inspector에 노출하고 싶으면
`RegisterReferenceFields(...)`를 같이 구현합니다.

예시:

```cpp
void MyComponent::RegisterReferenceFields(ReferenceFieldRegistry& registry)
{
    registry.RegisterGameObjectRef("Target Object", &m_targetObject);
    registry.RegisterComponentRef(
        "Target Rigidbody",
        reinterpret_cast<Component**>(&m_targetRigidbody),
        "Rigidbody2D");
}
```

이렇게 등록하면 Inspector에서 공통 방식으로 참조 필드가 그려집니다.

현재 지원되는 쪽은 아래와 같습니다.

- `GameObject` 참조 필드
- `Component` 참조 필드
- hierarchy에서 `GameObject` drag & drop
- 타입이 맞는 component 자동 연결
- SceneData 저장/로드


## 11. TimeScale과 DeltaTime 사용 기준

사용자 컴포넌트에서 시간을 다룰 때는 아래 기준으로 보면 됩니다.

- 기본 게임 로직은 `MainFrame::GetInstance()->DeltaTime()` 기준으로 구현합니다.
- `DeltaTime()`은 `TimeScale` 영향을 받습니다.
- 그래서 일반 게임 플레이 로직은 이 값을 쓰는 쪽이 맞습니다.

예를 들어 아래 같은 것들은 보통 `DeltaTime()`을 씁니다.

- 캐릭터 이동
- 점프 보정
- 일반 타이머
- 패턴 진행
- 스크롤 속도 반영

반대로 `MainFrame::GetInstance()->UnscaledDeltaTime()`은
`TimeScale` 영향을 받지 않습니다.

그래서 아래처럼 예외 연출에만 쓰는 편이 좋습니다.

- 잠깐 멈춤 연출 중에도 계속 돌아야 하는 회전
- 타임스케일과 무관하게 진행해야 하는 UI 효과
- 짧은 연출용 보정

정리하면:

- 기본 게임 로직: `DeltaTime()`
- 타임스케일 무시 연출: `UnscaledDeltaTime()`

최근 샘플 게임의 아이템 획득 연출도 이 기준을 따릅니다.
잠깐 `TimeScale = 0`으로 멈춘 뒤,
플레이어 회전과 UI 반응만 `UnscaledDeltaTime()`으로 진행하는 방식입니다.

## 12. 특정 Component만 받게 하고 싶을 때

`RegisterReferenceFields(...)`를 쓰다 보면
"Inspector에서는 특정 컴포넌트만 받게 하고 싶다"는 경우가 자주 생깁니다.

현재 프로젝트에서는 이때 아래 방식으로 정리합니다.

- Inspector에서 허용할 정확한 타입은 `expectedType` 문자열로 제한합니다.
- 실제 멤버 필드는 `Component*`로 둡니다.
- 실제 사용 시점에 `dynamic_cast`로 원하는 타입으로 꺼냅니다.

예를 들면 아래처럼 씁니다.

```cpp
Component* m_poolComponent = nullptr;

void MyComponent::RegisterReferenceFields(ReferenceFieldRegistry& registry)
{
    registry.RegisterComponentRef(
        "ObjectPool Component",
        &m_poolComponent,
        "SpawnObjectPoolComponent");
}
```

그리고 사용할 때는 이렇게 꺼냅니다.

```cpp
SpawnObjectPoolComponent* pool =
    dynamic_cast<SpawnObjectPoolComponent*>(m_poolComponent);
```

이유는 현재 `RegisterComponentRef(...)`가 `Component**`를 받기 때문입니다.
그래서 `SpawnObjectPoolComponent**` 같은 값을 직접 넘기는 방식은 현재 구조와 맞지 않습니다.

정리하면 지금 기준은 아래처럼 보면 됩니다.

- Inspector에서 타입 검사: `expectedType`
- 실제 저장: `Component*`
- 실제 사용: `dynamic_cast`

## 13. 한 줄 정리

사용자 확장은 지금 기준으로
`KirbyGameDll` 안에 컴포넌트/액션을 만들고,
registry에 등록해서 에디터와 SceneData에 연결하는 방식으로 이해하면 됩니다.
