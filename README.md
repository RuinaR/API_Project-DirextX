# API_Project-DirextX

WinAPI 기반으로 제작했던 2D 게임 프로젝트의 렌더링 방식을 DirectX로 확장/개선한 프로젝트입니다.  
단순 게임 제작이 아니라, **엔진 레이어와 게임 레이어를 분리**하고  
렌더링, 카메라, 충돌, 애니메이션, 플레이어 액션 등의 구조를 직접 구현해보는 것을 목표로 진행했습니다.

---

## 프로젝트 소개

이 프로젝트는 기존에 제작했던 API 기반 게임 프로젝트를 발전시켜,  
렌더링 방식을 DirectX로 변경하고 외부 SDK를 적용하면서 구조를 개선한 프로젝트입니다.

게임 플레이 기능을 구현하는 것에만 집중하지 않고,  
**엔진 프레임워크와 게임 로직을 분리하는 구조**를 직접 만들어보며  
게임 클라이언트 프로그래머로서 필요한 기초 역량을 쌓는 데 초점을 두었습니다.

---

## 개발 목표

- WinAPI 기반 프로젝트를 DirectX 기반으로 확장
- 렌더링 처리 구조 개선
- 게임 로직과 엔진 레이어 분리
- 플레이어, 몬스터, 충돌, 카메라, 애니메이션 흐름 직접 구현

---

## 개발 환경

- **Language**: C++
- **Graphics API**: DirectX
- **Framework**: WinAPI 기반 구조 포함
- **IDE**: Visual Studio
- **Version Control**: Git / GitHub

## 프로젝트 구조

```text
Solution_Kirby
├─ EngineFramework                    # 엔진 공통 기능
│  ├─ AnimationManager               # 애니메이션 관리
│  ├─ BitmapRender                   # 렌더링 처리
│  ├─ Buffer                         # 버퍼 관련 처리
│  ├─ Camera                         # 카메라 제어
│  ├─ Collider / BoxCollider         # 충돌 판정
│  ├─ GameObject                     # 게임 오브젝트 공통 기반
│  ├─ Scene                          # 씬 공통 처리
│  └─ 기타 엔진 유틸리티
│
├─ KirbyGame                         # 실제 게임 플레이 로직
│  ├─ Player                         # 플레이어 이동, 점프, 공격
│  ├─ MonsterAI                      # 몬스터 AI
│  ├─ GameScene                      # 게임 씬 구성
│  ├─ SceneChanger                   # 씬 전환 처리
│  ├─ LandChecker                    # 지형 / 바닥 판정
│  └─ 기타 게임 오브젝트 및 로직
│
├─ External SDK
│  ├─ Box2D                          # 물리 및 충돌 처리
│  ├─ ImGui                          # 디버그 UI 및 툴용 인터페이스
│  └─ FBX SDK                        # FBX 애니메이션 / 모델 데이터 처리
│
├─ Resources                         # 이미지, 애니메이션, 기타 리소스
└─ Solution 파일 및 빌드 관련 폴더
```

### EngineFramework
엔진 성격의 공통 기능을 모아둔 영역입니다.


### KirbyGame
실제 게임 플레이 로직을 담당하는 영역입니다.


엔진 기능과 게임 오브젝트/플레이 로직을 분리하려고 노력했습니다.

## 구현하면서 중점적으로 고민한 점

### 1. 엔진 레이어와 게임 레이어 분리
게임 로직만 빠르게 구현하면 구조가 금방 복잡해지기 때문에,  
공통 기능은 EngineFramework로 분리하고 게임별 로직은 KirbyGame에서 다루도록 구성했습니다.

### 2. 플레이어 액션의 세분화
이동, 점프, 기본 공격, 검 공격, 충돌 등 플레이어 관련 기능을 세분화하여  
각 동작을 독립적으로 관리하려고 했습니다.  
이 과정에서 액션 처리와 충돌 처리의 연결 방식도 직접 구현하며 흐름을 이해할 수 있었습니다.

### 3. 렌더링 방식 확장 경험
기존 API 프로젝트에서 더 나아가 DirectX 기반으로 렌더링 방식을 확장하면서  
그래픽 처리 방식과 프로젝트 구조 변화가 실제 코드에 어떤 영향을 주는지 경험할 수 있었습니다.

---

## 어려웠던 점

- 렌더링, 충돌, 애니메이션, 입력 처리가 서로 연결되어 있어 기능이 늘어날수록 구조가 복잡해졌습니다.
- 플레이어 클래스에 여러 책임이 몰리기 쉬워, 기능을 분리하면서도 기존 흐름을 유지하는 것이 쉽지 않았습니다.
- 외부 SDK 적용과 빌드 환경 구성 과정에서 시행착오가 있었습니다.

---

## 아쉬웠던 점

- README와 구조 설명이 부족해 처음 보는 사람이 프로젝트 의도를 바로 이해하기 어렵습니다.
- 일부 클래스는 기능이 커지면서 책임이 많이 모인 부분이 있습니다.
- 빌드 산출물, IDE 관련 파일 등 저장소 정리가 더 필요합니다.

## 개선하고 싶은 점

- Player 및 일부 시스템 클래스의 책임 재분리
- 이벤트 기반 구조 또는 상태 기반 구조로 리팩터링
- 빌드 산출물 / 사용자 환경 파일 정리
- 실행 영상, 구조도, 클래스 관계 설명 추가
- 렌더링 흐름과 게임 루프 설명 문서화

---

## 배운 점

이 프로젝트를 통해 단순히 게임 기능을 구현하는 수준을 넘어,  
**게임 로직과 엔진 구조를 어떻게 분리할 것인지**,  
그리고 렌더링/충돌/애니메이션/입력이 연결되는 흐름을 어떻게 설계할 것인지 고민할 수 있었습니다.

또한 C++ 기반 프로젝트에서 클래스 구조와 책임 분리가 얼마나 중요한지 체감했고,  
추후 Unity나 다른 게임 클라이언트 환경에서도 구조적인 사고를 적용할 수 있는 기반이 되었습니다.
