# Kirby Engine 프로젝트 기술/사용 설명서

## 1. 문서 소개

이 문서는 `Solution_Kirby` 프로젝트를 처음 받았을 때, 전체 구조를 빠르게 이해하고 직접 빌드, 실행, 수정까지 해볼 수 있도록 정리한 기술/사용 설명서입니다.  
가능한 한 실제 폴더 구조와 코드 흐름을 기준으로 설명하려고 했고, "이 프로젝트가 정확히 뭐고, 어디서부터 봐야 하는지"가 헷갈리지 않도록 자세하게 적었습니다.

이 프로젝트는 단순히 게임 하나만 있는 구조가 아니라,

- 공용 엔진 역할을 하는 `EngineFramework`
- 실제 게임 로직이 들어 있는 `KirbyGameDll`
- 게임 실행용 프로그램인 `BuildResultGame`
- 에디터 실행용 프로그램인 `BuildEdit`
- 외부 라이브러리인 `Box2D`, `Imgui`, `FBX SDK`

이렇게 여러 프로젝트가 같이 묶여 있는 형태입니다.

즉, 이 프로젝트는 "카비 스타일의 2D/2.5D 게임을 만들기 위한 자체 엔진 + 에디터 + 게임 DLL 구조"라고 이해하면 가장 편합니다.

---

## 2. 프로젝트 한 줄 요약

이 프로젝트는 **플러그인 방식으로 게임 DLL을 로드해서**,  
같은 엔진 프레임워크를 기반으로

- `BuildEdit`에서는 씬 편집기처럼 동작하고
- `BuildResultGame`에서는 실제 게임처럼 동작하는

구조를 가진 Windows C++ 게임 엔진 프로젝트입니다.

핵심 포인트는 둘 다 결국 `KirbyGameDll.dll`을 불러와서 실행한다는 점입니다.  
차이는 초기화할 때 `Edit` 모드로 들어가느냐, `Game` 모드로 들어가느냐입니다.

## 4. 솔루션 내부 프로젝트 역할

### 4-1. `EngineFramework`

이 프로젝트는 공용 엔진의 핵심입니다.

대략적으로 아래 역할을 담당합니다.

- 윈도우 생성과 메인 루프 관리
- DirectX9 기반 렌더링
- ImGui 기반 에디터 UI
- Box2D 물리 처리
- 오브젝트/컴포넌트 관리
- 씬 로드/저장
- 에디터 모드와 게임 모드 분기 처리

코드를 보면 `MainFrame`, `WindowFrame`, `ObjectManager`, `SceneDataManager` 같은 클래스들이 중심 역할을 합니다.

특히 `MainFrame.h`에는 아래처럼 렌더 모드가 정의되어 있습니다.

- `RenderType::Edit`
- `RenderType::Game`

이 모드 차이에 따라 같은 엔진이라도 에디터처럼 동작할지, 게임처럼 동작할지가 나뉘게 됩니다.

### 4-2. `KirbyGameDll`

이 프로젝트는 실제 게임 내용이 들어 있는 DLL입니다.  
쉽게 말하면 엔진 위에 올라가는 "게임 플러그인"입니다.

주요 역할은 다음과 같습니다.

- `IPlugin` 인터페이스 구현
- 시작 씬 결정
- 사용자 컴포넌트 등록
- 사용자 UI 액션 등록
- 게임 시작 시 엔진과 씬 연결

`KirbyGame.cpp`를 보면 초기화 과정에서

1. 엔진 기본 컴포넌트를 등록하고
2. 게임 전용 컴포넌트를 등록하고
3. 게임 전용 UI 액션을 등록하고
4. 시작 씬 이름을 결정한 뒤
5. `MainFrame`을 생성하고 실행

하는 흐름으로 진행됩니다.

### 4-3. `BuildEdit`

이 프로젝트는 에디터 실행 프로그램입니다.  
실행하면 `KirbyGameDll.dll`을 로드한 뒤 `RenderType::Edit` 모드로 초기화합니다.

즉, 별도의 게임 코드를 다시 작성하는 것이 아니라, 같은 DLL을 "편집 모드"로 실행하는 구조입니다.

이 모드에서는 보통 아래 작업을 한다고 보면 됩니다.

- 씬 열기
- 오브젝트 생성/수정
- Inspector에서 컴포넌트 값 편집
- SceneData 저장
- BuildSettings 편집
- 최종 빌드 출력 생성

### 4-4. `BuildResultGame`

이 프로젝트는 실제 게임 실행 프로그램입니다.  
마찬가지로 `KirbyGameDll.dll`을 로드하지만 이번에는 `RenderType::Game`으로 초기화합니다.

즉, 에디터에서 만든 씬과 설정을 바탕으로 실제 플레이용 프로그램처럼 동작합니다.

### 4-5. `Box2D`, `Imgui`

- `Box2D`: 물리 계산용 라이브러리
- `Imgui`: 에디터 UI 구성용 라이브러리

이 둘은 엔진이 사용하는 외부 라이브러리라고 보면 됩니다.

## 5. 실행 구조 이해하기

이 프로젝트의 실행 흐름은 생각보다 깔끔합니다.

### 5-1. 공통 진입 흐름

`BuildEdit/main.cpp`와 `BuildResultGame/main.cpp`는 둘 다 거의 같은 구조입니다.

1. `BuildLoopCls` 객체를 만든다.
2. `Init()`에서 `KirbyGameDll.dll`을 로드한다.
3. DLL 안의 플러그인 객체를 가져온다.
4. `Initialize()`를 호출한다.
5. `while` 루프에서 `AllUpdate()`를 반복 호출한다.
6. 종료 시 `AllRelease()` 후 DLL을 언로드한다.

### 5-2. 에디터와 게임의 차이

두 실행 파일의 가장 중요한 차이는 아래 한 줄입니다.

- `BuildEdit`는 `RenderType::Edit`
- `BuildResultGame`는 `RenderType::Game`

즉, 실행 파일은 껍데기 역할만 하고, 실제 동작 차이는 DLL 초기화 인자에서 갈립니다.

이 구조의 장점은

- 엔진 재사용이 쉽고
- 게임/에디터가 같은 데이터를 공유하며
- 기능 추가 시 DLL 중심으로 관리하기 편하다는 점입니다.

## 6. 씬 데이터 구조

이 프로젝트는 씬 정보를 `SceneData/*.json` 파일로 저장합니다.

예를 들어 현재 폴더에는 아래 파일이 있습니다.

- `SceneData/TestScene1.json`
- `SceneData/TestScene2.json`

JSON 안에는 대략 아래 정보가 저장됩니다.

- 씬 버전
- 씬 이름
- 카메라 위치/회전/투영 설정
- 오브젝트 목록
- 부모-자식 관계
- Transform 정보
- 각 오브젝트가 가진 컴포넌트 정보

즉, Unity의 Scene 저장 파일을 단순화한 느낌으로 이해하면 편합니다.

### 6-1. 예시로 보이는 컴포넌트 종류

샘플 씬을 보면 아래 같은 컴포넌트들이 등장합니다.

- `ImageRender`
- `AnimationRender`
- `FBXRender`
- `UIImage`
- `UILabel`
- `UIButton`
- `SampleSpinComponent`

즉, 엔진 기본 컴포넌트와 사용자 정의 컴포넌트가 함께 JSON에 기록되는 구조입니다.

### 6-2. 저장 방식 특징

이 프로젝트는 컴포넌트 전체를 바이너리로 저장하는 것이 아니라,  
각 컴포넌트가 `Serialize()` / `Deserialize()`를 구현해서 JSON 문자열 형태로 저장/복원됩니다.

그래서 새 컴포넌트를 만들 때 가장 중요한 부분 중 하나가

- 저장 가능한 값만 JSON에 넣기
- 런타임 포인터 같은 값은 저장하지 않기

입니다.

## 7. BuildSettings 설명

`BuildSettings.json`은 어떤 씬을 포함해서 빌드할지, 그리고 게임 시작 시 어떤 씬을 먼저 열지 정하는 파일입니다.

현재 파일 내용은 아래 의미를 가집니다.

- 시작 씬: `TestScene2`
- 빌드 포함 씬: `TestScene1`, `TestScene2`

이 파일은 단순해 보이지만 꽤 중요합니다.  
왜냐하면 게임 실행 시 시작 씬 결정에 직접 사용되기 때문입니다.

### 7-1. 시작 씬 결정 방식

코드 흐름상 시작 씬은 아래 순서로 결정됩니다.

1. `BuildSettings.json`에서 `startScene`을 읽는다.
2. 읽기에 실패하면 기본값을 쓴다.
3. `Edit` 모드 기본값은 `NewScene`
4. `Game` 모드 기본값은 `StartScene`

즉, `BuildSettings.json`이 있으면 그 값을 우선 사용하고, 없거나 잘못되면 모드별 기본 이름으로 대체됩니다.

### 7-2. 왜 중요한가

만약 게임 모드에서 시작 씬 파일이 없으면, 코드상 메시지 박스를 띄우고 초기화에 실패할 수 있습니다.  
그래서 새 프로젝트를 받을 때는 `BuildSettings.json`과 `SceneData` 폴더를 같이 확인하는 습관이 필요합니다.

## 8. 빌드 결과물과 출력 폴더

이 프로젝트는 `Directory.Build.props`에서 공통 출력 폴더를 맞춰 두었습니다.

출력 경로 형식은 아래와 같습니다.

`Bin/$(Configuration)_$(Platform)`

예를 들면,

- `Bin/Debug_Win32/`
- `Bin/Debug_x64/`
- `Bin/Release_Win32/`

같은 형태로 결과물이 정리됩니다.

이렇게 해 둔 장점은 프로젝트별 출력물이 흩어지지 않고, 실행 파일과 DLL을 한 곳에서 확인하기 쉽다는 점입니다.

### 8-1. 빌드 후 자동 복사되는 것들

`Directory.Build.targets`를 보면 애플리케이션 빌드 후 아래 자원들이 출력 폴더로 복사됩니다.

- `Debug/Bitmaps`
- `Debug/Map`
- `Debug/FBX`
- `libfbxsdk.dll`
- 필요 시 `D3DX9_43.dll`

즉, 실행 파일만 생성되는 것이 아니라 런타임에 필요한 자원도 같이 배치되도록 설정되어 있습니다.

이 부분이 없으면 exe는 생성되어도 텍스처나 FBX 파일을 못 찾아서 실행이 깨질 가능성이 큽니다.

## 9. 개발 환경 및 의존성

기본적으로 이 프로젝트는 **Visual Studio C++ 솔루션** 기준으로 만들어져 있습니다.

필요한 요소는 대략 아래와 같습니다.

- Visual Studio 2022 계열
- MSBuild
- Windows SDK
- C++ 데스크톱 개발 환경
- DirectX 9 관련 헤더/라이브러리

### 9-1. DirectX SDK / D3DX 처리 방식

이 프로젝트는 오래된 `D3DX` 의존성이 있기 때문에, 환경에 따라 설정이 귀찮을 수 있습니다.  
다행히 현재 구조에서는 이 부분을 어느 정도 자동으로 처리하도록 해 둔 상태입니다.

코드를 보면 다음 방식으로 처리합니다.

1. 우선 `External/packages/Microsoft.DXSDK.D3DX.9.29.952.8` 패키지를 확인
2. 없으면 설치된 `Microsoft DirectX SDK (June 2010)` 경로를 확인
3. 해당 include/lib/bin 경로를 빌드 설정에 연결

즉, 예전 방식처럼 무조건 전역 SDK를 깔아야만 하는 구조는 아니고,  
NuGet 기반 복구 경로도 같이 고려해 둔 형태입니다.

### 9-2. FBX SDK

프로젝트 루트의 `sdk.txt`에는 Autodesk FBX SDK 관련 링크가 적혀 있습니다.  
폴더 안에도 `fbxsdk/`, `Lib/`, `include/`가 함께 있어서 FBX 렌더링/로딩 기능에 사용되는 것으로 보입니다.

## 10. 실제 사용 방법

### 10-1. Visual Studio에서 실행하는 방법

가장 무난한 방법은 Visual Studio에서 `Solution_Kirby.sln`을 여는 것입니다.

추천 순서는 아래와 같습니다.

1. `Solution_Kirby.sln` 열기
2. 플랫폼을 `Win32` 또는 `x64` 중 하나로 맞추기
3. 구성은 우선 `Debug`로 두기
4. `BuildEdit`를 시작 프로젝트로 설정해서 실행
5. 씬 편집 및 저장 확인
6. 이후 `BuildResultGame`를 실행해서 실제 게임 모드 확인

처음 확인할 때는 에디터부터 켜 보는 것이 더 좋습니다.  
왜냐하면 현재 어떤 씬이 잡혀 있는지, 오브젝트와 컴포넌트가 어떻게 저장되어 있는지 직접 눈으로 확인할 수 있기 때문입니다.

### 10-2. VSCode에서 빌드하는 방법

루트에 있는 `VSCode_MSBuild.md`를 보면 VSCode에서도 기존 `.sln`, `.vcxproj`를 유지한 채 MSBuild로 빌드할 수 있게 되어 있습니다.

문서상 핵심은 아래와 같습니다.

- 폴더 자체를 VSCode로 연다.
- MSBuild 기반 task를 사용한다.
- 출력은 `Bin/...` 아래로 정리된다.
- D3DX 패키지가 필요하면 스크립트가 복구를 시도한다.

즉, VSCode를 써도 되지만, 처음 구조를 파악할 때는 Visual Studio가 더 편할 가능성이 높습니다.

### 10-3. 어떤 실행 파일을 언제 쓰는가

- 씬 수정/저장/빌드 설정 변경: `BuildEdit`
- 실제 플레이 확인: `BuildResultGame`

이 둘을 혼동하지 않는 것이 중요합니다.

## 11. 씬 작업 방법

### 11-1. 씬 생성

에디터 모드에서는 기본적으로 새 씬을 만들 수 있고, 필요하면 `NewScene` 같은 이름으로 초기 생성됩니다.

### 11-2. 씬 열기

`SceneData` 폴더 안의 JSON 파일 목록을 기반으로 열 수 있습니다.

예를 들면 현재는 아래 씬들이 있습니다.

- `TestScene1`
- `TestScene2`

### 11-3. 씬 저장

에디터에서 현재 씬을 수정한 뒤 저장하면 `SceneData/*.json` 파일이 갱신됩니다.

코드상으로는 씬의 이전 스냅샷을 잡아두고, 현재 상태와 비교해서 dirty 여부를 판단하는 구조도 들어 있습니다.  
그래서 수정 사항이 있는지 없는지를 내부적으로 추적하려는 의도가 보입니다.

### 11-4. 씬 이름 주의

씬 이름은 파일명과 연결되므로 너무 특수문자가 많거나 잘못된 이름을 쓰면 문제가 생길 수 있습니다.  
프로젝트를 유지보수하려면 영문/숫자 중심의 간단한 이름을 쓰는 것이 안전합니다.

## 12. 사용자 확장 기능

이 프로젝트의 재미있는 부분은 사용자 확장 구조가 어느 정도 정리되어 있다는 점입니다.

### 12-1. 사용자 컴포넌트 추가 방법

기본 흐름은 아래와 같습니다.

1. `KirbyGameDll/UserComponents/Scripts`에 새 `.h`, `.cpp` 파일 작성
2. `Component`를 상속
3. `kComponentType` 같은 직렬화용 타입 이름 정의
4. `GetSerializableType()` 반환값 구현
5. `DrawInspector()` 구현
6. `Serialize()`, `Deserialize()` 구현
7. `UserComponents/GameComponentRegistry.cpp`에 등록

중요한 점은 **핫 리로드가 지원되지 않는다는 것**입니다.  
즉, C++ 코드 추가 후에는 그냥 저장만 해서는 반영되지 않고, DLL을 다시 빌드하고 에디터를 재실행해야 합니다.

### 12-2. 사용자 액션 추가 방법

UI 버튼 액션 등도 문자열 키 기반으로 연결됩니다.

대략 흐름은 아래와 같습니다.

1. `KirbyGameDll/UserActions/Scripts`에 액션 코드 작성
2. `UserActions/GameActionRegistry.cpp`에 액션 키 등록
3. 에디터에서 `UIButton`의 `actionKey`로 연결
4. SceneData에는 함수 포인터 대신 문자열 키만 저장

이 구조는 JSON 저장에 유리합니다.  
콜백 자체를 저장하는 것이 아니라 `"SampleButtonAction"` 같은 이름만 저장하고, 실행 시 registry에서 다시 연결하기 때문입니다.

## 16. 주의사항 및 느낀 점

### 16-1. DLL 구조라서 재실행이 자주 필요함

이 프로젝트는 `KirbyGameDll.dll` 중심 구조라서, 게임 로직을 수정하면 에디터/게임 exe를 다시 띄워야 하는 경우가 많습니다.

### 16-2. 구형 그래픽 API 의존성

DirectX9 + D3DX 계열 의존성이 있어서 최신 환경에서는 세팅 이슈가 생길 수 있습니다.  
다만 현재 스크립트와 공통 빌드 설정에서 이 부분을 꽤 많이 보완해 둔 상태입니다.

### 16-3. SceneData가 중요 자산임

코드보다도 실제 작업 결과는 `SceneData` JSON에 많이 담기기 때문에,  
씬 파일 백업을 자주 해두는 것이 좋습니다.

### 16-4. 엔진 코드와 게임 코드를 구분해서 보는 것이 좋음

모든 코드를 한 번에 이해하려고 하면 너무 복잡할 수 있습니다.  
그래서 아래처럼 나눠서 보는 것이 좋습니다.

- 엔진 공통: `EngineFramework`
- 게임별 구현: `KirbyGameDll`
- 실행기: `BuildEdit`, `BuildResultGame`

## 17. 결론

정리하면 이 프로젝트는 단순한 한 개의 게임 실행 파일이 아니라,  
**엔진 프레임워크 + 게임 DLL + 에디터 + 실행기 + 씬 데이터 시스템**이 합쳐진 형태의 비교적 잘 분리된 구조를 가지고 있습니다.

학생 프로젝트 느낌으로 보자면,

- 직접 엔진 구조를 설계해 보려는 시도
- 씬 직렬화(JSON) 시스템 구현
- 에디터/게임 모드 분리
- 사용자 확장 포인트 제공

이런 요소들이 잘 들어가 있는 편입니다.

처음에는 폴더가 많아서 복잡해 보이지만, 핵심만 잡으면 구조는 아래 한 문장으로 정리됩니다.

**`BuildEdit`와 `BuildResultGame`이 `KirbyGameDll.dll`을 로드하고, `EngineFramework`가 실제 실행을 담당하며, `SceneData`와 `BuildSettings`가 프로젝트 내용을 결정한다.**

이 문서를 시작점으로 삼아서,

- 먼저 에디터를 켜 보고
- 씬을 저장해 보고
- DLL 쪽 컴포넌트를 하나 수정해 보고
- 마지막으로 게임 모드 실행까지 확인

해보면 프로젝트 구조가 훨씬 자연스럽게 익숙해질 것 같습니다.

## 18. 최근 구조 갱신 (2026-04-30)

이 문서의 앞부분은 프로젝트의 큰 구조 설명에 집중되어 있고, 최근 구현된 `Physics2D`, `Rigidbody2D`, `Raycast`, `참조 필드` 관련 내용은 아래처럼 이해하면 됩니다.

### 18-1. 현재 엔진 구조 방향

장기적으로는 다음 분리를 목표로 하고 있습니다.

- `EngineRuntime`: 게임 실행에 필요한 공통 기능
- `EngineEditor`: ImGui, Hierarchy, Inspector, BuildSettings 같은 편집 기능
- `KirbyGameDll`: 게임별 `Component`, `Action`, bootstrap
- 에디터 실행 경로: Runtime + Editor + GameDll
- 게임 실행 경로: Runtime + GameDll

현재 저장소에서는 이 역할이 아직 `EngineFramework` 하나 안에 많이 같이 들어 있지만, 구현 방향은 위 구조를 향해 가고 있습니다.

### 18-2. Physics2D 기준 정리

현재 물리는 3D가 아니라 **2D 전용** 방향으로 정리 중입니다.

- 기존 `BoxCollider` 저장 type은 이제 canonical 기준으로 `BoxCollider2D`
- legacy SceneData 호환을 위해 `BoxCollider` alias 로드는 유지
- Add Component UI에는 `BoxCollider2D`만 노출
- `Rigidbody2D` 컴포넌트가 추가되어 body type, damping, gravity, velocity, collision detection, interpolate 등을 관리
- `Rigidbody2D`가 있으면 collider는 그 body에 fixture를 붙이고, 없으면 collider가 static fallback body를 유지

즉 현재는 완전한 `Collider2D / Rigidbody2D` 최종 분리 전의 **호환 단계**라고 보면 됩니다.

### 18-3. Rigidbody2D와 Collider 현재 동작 방식

최근 안정화된 핵심은 다음과 같습니다.

- `Collider -> Rigidbody2D` 순서로 붙여도 동작
- `Rigidbody2D -> Collider` 순서로 붙여도 동작
- `Rigidbody2D` body 재생성 시 attached collider fixture를 다시 부착
- `Rigidbody2D`만 삭제하면 collider는 static fallback body로 복귀
- `Collider`만 삭제하면 rigidbody body는 유지되고 fixture만 제거
- `GameObject` 삭제 중에는 fallback body를 다시 만들지 않게 방지
- dynamic body 상태에서 GameObject position/rotation을 직접 바꿔도 body를 다시 깨워서 시뮬레이션이 계속되게 보강

### 18-4. Collision callback 정리

예전에는 collision callback이 `body userData`를 기준으로 `BoxCollider*`를 읽는 구조라, body 하나에 fixture 여러 개가 붙는 구조에 취약했습니다.

지금은 다음처럼 바뀌었습니다.

- collider fixture 생성 시 `fixture userData`에 `Collider*` 저장
- `CollisionListener`는 `fixture userData`에서 `Collider*`를 읽음
- `BeginContact`, `EndContact`, `PreSolve` 모두 fixture 기준으로 `OnCollisionEnter/Exit/Stay` 호출

이 구조는 이후 `CircleCollider2D` 같은 다른 collider shape를 추가할 때도 더 자연스럽게 확장됩니다.

### 18-5. Ray / Mouse / Physics2D Raycast

입력과 피킹도 함께 정리되었습니다.

- UI 입력은 계속 `Mouse::GetGameViewPos()` 기반 screen-space 판정 유지
- `Mouse::ScreenPointToRay()`가 추가되어 현재 카메라 기준 3D ray 생성 가능
- `Math/Ray.h` 추가
- `Physics2D::Raycast(const Ray&, ...)` 최소 구현 추가
- 현재 raycast는 `ObjectManager` 순회 + `BoxCollider` AABB 판정 기반
- 에디터 모드에서 ray debug 텍스트와 debug line 시각화 가능

즉 UI 입력과 월드 raycast 입력이 분리되기 시작한 상태입니다.

### 18-6. GameObject Reference Field / Resolve

Inspector에서 다른 오브젝트를 참조하는 기본 흐름도 들어가 있습니다.

- Hierarchy drag/drop payload 이름은 `HierarchyGameObject`
- `EditorInspectorWindow::DrawGameObjectReferenceField(...)` helper 제공
- SceneData에는 raw pointer 대신 `objectId`만 저장
- 로드 후 `objectId -> GameObject*` map으로 `ResolveReferences(...)` 호출
- 샘플로 `SampleSpinComponent`에 `targetObjectId` 저장/복원 패턴 적용

이 구조는 이후 `follow target`, `button target`, `trigger target` 같은 참조형 컴포넌트에 재사용할 수 있습니다.

### 18-7. 현재 문서와 코드의 차이 주의

이 문서의 앞부분 설명만 보면 collider가 body를 직접 들고 있는 예전 구조로 읽힐 수 있지만, 현재 코드는 이미 `Rigidbody2D` 도입을 시작한 중간 단계입니다.

따라서 최신 코드를 볼 때는 다음처럼 이해하는 것이 맞습니다.

- 저장 이름 기준: `BoxCollider2D`, `Rigidbody2D`
- 내부 구현 이름 기준: 아직 `Collider`, `BoxCollider`
- 최종 목표: `Rigidbody2D`가 body 소유, `Collider2D`가 fixture/shape 담당

## 19. 최신 빌드/실행 설정 메모 (2026-05-01)

최근 환경 점검 기준으로, 이 프로젝트는 **Visual Studio 2022 + Windows SDK + DirectX 9(D3DX) + FBX SDK 런타임** 조합에서 정상 빌드/실행을 확인했습니다.

이번 점검에서 특히 중요했던 것은 다음 두 흐름입니다.

- VS Code의 `실행 및 디버그` 버튼으로 `BuildEdit` / `BuildResultGame`을 빌드 후 실행하는 흐름
- 에디터 내부 `Build Game` 버튼이 `Release|x64` 게임 빌드 후 `BuildOutput` 패키지를 만드는 흐름

둘 다 되게 하려면, 단순히 코드가 맞는 것만으로는 부족하고 **PowerShell / MSBuild / D3DX 런타임 탐색 방식**이 환경에 따라 흔들리지 않게 정리되어 있어야 합니다.

### 19-1. VS Code 실행/디버그 설정

`.vscode/launch.json`에는 아래 실행 구성이 들어 있습니다.

- `Debug Game (Win32)`
- `Debug Editor (Win32)`
- `Debug Game (x64)`
- `Debug Editor (x64)`

실제 실행 파일은 `Bin/Debug_Win32`, `Bin/Debug_x64` 아래 exe를 직접 가리킵니다.

문제는 `preLaunchTask`가 연결된 `.vscode/tasks.json` 쪽이었습니다.  
일부 환경에서는 `powershell` 명령이 PATH에 없어서, VS Code에서 실행 버튼을 눌러도 빌드 전에 바로 실패할 수 있었습니다.

그래서 현재는 task의 `command`를 단순 `powershell`이 아니라 아래 Windows 기본 경로 기반으로 두는 것이 안전합니다.

- `${env:windir}\\System32\\WindowsPowerShell\\v1.0\\powershell.exe`

이 방식의 장점:

- `powershell`이 PATH에 등록된 기존 PC에서도 그대로 동작
- PATH가 비어 있거나 다른 셸만 잡힌 PC에서도 동작
- 프로젝트 루트에서 `Scripts/Invoke-MSBuild.ps1`를 일관되게 실행 가능

즉 VS Code 쪽은 **"별칭에 의존하지 않고 Windows 기본 PowerShell 실행 파일을 직접 호출"** 하는 것이 호환성 면에서 가장 안전합니다.

### 19-2. `Scripts/Invoke-MSBuild.ps1` 역할

VS Code task에서 실제로 호출하는 스크립트는 `Scripts/Invoke-MSBuild.ps1`입니다.

이 스크립트는 다음 순서로 동작합니다.

1. `vswhere.exe`로 최신 Visual Studio의 `MSBuild.exe` 위치를 찾음
2. 없으면 `MSBuild.exe` 이름으로 직접 호출 시도
3. `DXSDK_DIR`가 없으면:
   - 먼저 설치형 `Microsoft DirectX SDK (June 2010)`을 확인
   - 그것도 없으면 `Microsoft.DXSDK.D3DX` NuGet 패키지를 복구
4. 이후 `.sln` 기준으로 원하는 `Configuration`, `Platform`, `Target`을 빌드

즉 VS Code 빌드 흐름은 비교적 현대적인 방식으로 이미 잘 정리되어 있고, 문제는 주로 **task에서 어떤 PowerShell을 호출하느냐** 쪽이었습니다.

### 19-3. 에디터의 `Build Game` 흐름

에디터의 `Build Settings` 패널에 있는 `Build Game` 버튼은 VS Code task를 호출하지 않습니다.  
이 버튼은 `EngineFramework/Editor/EditorBuildSettingsPanel.cpp` 안에서 별도의 비동기 빌드 흐름을 사용합니다.

핵심 단계는 다음과 같습니다.

1. 현재 씬이 dirty 상태면 먼저 저장
2. `BuildSettings.json` 저장
3. `Tools/MSBuild/msbuild.cmd`를 통해 `Release|x64`의 `BuildResultGame` 빌드
4. `BuildOutput` 폴더 준비
5. 빌드 대상 `SceneData`만 `BuildOutput/SceneData`로 복사
6. `BuildOutput/BuildSettings.json` 생성
7. `Release_x64`의 `BuildResultGame.exe`, `KirbyGameDll.dll`, 기타 DLL 복사
8. 실제 씬에서 참조하는 `Resources`만 `BuildOutput/Resources`로 최소 복사

즉 이 기능은 단순 복사가 아니라:

- 실제 `Release|x64` 게임 빌드 실행
- 시작 씬/포함 씬 검증
- 필요한 SceneData/Resources만 추려서 패키징

까지 포함하는 **실제 배포용 패키징 흐름**에 가깝습니다.

### 19-4. `Build Game`에서 실제로 문제였던 부분

이번 점검에서 확인한 가장 큰 이슈는 `Tools/MSBuild/msbuild.cmd`였습니다.

이 스크립트는 원래:

- `where msbuild`
- 실패 시 `C:\Program Files\Microsoft Visual Studio\17\...`
- 또는 `18\...`

같은 고정 경로만 확인하고 있었습니다.

그런데 실제 Visual Studio 2022 기본 설치 경로는 보통 아래 형태입니다.

- `C:\Program Files\Microsoft Visual Studio\2022\Community\MSBuild\Current\Bin\MSBuild.exe`

즉 `17`, `18` 폴더명만 보는 방식은 일부 환경에서는 우연히 맞아도, 일반적인 2022 설치 환경에서는 `Build Game`이 **MSBuild를 못 찾고 바로 실패**할 수 있습니다.

그래서 현재는 `msbuild.cmd`가 아래 순서로 찾는 구조가 더 안전합니다.

1. `where msbuild`
2. `Visual Studio\2025\...`
3. `Visual Studio\2022\...`
4. 마지막 fallback으로 기존 `17`, `18` 경로

이렇게 해두면:

- PATH에 `msbuild`가 잡힌 기존 개발 PC는 그대로 동작
- 일반적인 Visual Studio 2022 설치 PC도 동작
- 향후 2025 계열 경로도 일부 대응 가능

즉 에디터 내부 `Build Game`은 **VS Code 설정과 별개로 `msbuild.cmd` 자체 호환성이 중요**합니다.

### 19-5. D3DX9_43.dll 복사 이슈

또 하나 중요한 점은 `D3DX9_43.dll`입니다.

이 프로젝트는 `Directory.Build.props` / `Directory.Build.targets`를 통해 빌드 후 런타임 파일을 `Bin/<Config>_<Platform>`으로 복사합니다.  
그런데 원래 설정은 **NuGet으로 복구된 D3DX 패키지 경로**만 `D3DXBinaryDir`로 잡고 있어서, 설치형 DirectX SDK 또는 시스템 런타임만 있는 PC에서는 `D3DX9_43.dll`이 출력 폴더에 빠질 수 있었습니다.

이 상태에서는:

- 현재 개발 PC처럼 시스템에 이미 `D3DX9_43.dll`이 있으면 실행이 됨
- 하지만 `BuildOutput`을 다른 PC로 옮겼을 때는 런타임 누락으로 실패 가능

그래서 현재는 `D3DXBinaryDir` fallback도 함께 고려하는 것이 안전합니다.

- `Win32`: `%SystemRoot%\\SysWOW64\\D3DX9_43.dll`
- `x64`: `%SystemRoot%\\System32\\D3DX9_43.dll`

이렇게 하면:

- NuGet 복구 환경에서도 복사 가능
- 설치형 DXSDK/시스템 런타임 환경에서도 복사 가능
- 최종 `BuildOutput`이나 `Bin/Release_x64`에 필요한 D3DX DLL이 같이 따라오게 됨

### 19-6. 이번 점검 기준 실제 확인 결과

최근 점검에서 직접 확인한 내용은 다음과 같습니다.

- `Debug|x64` 솔루션 빌드 성공
- `Release|x64` 솔루션 빌드 성공
- `BuildEdit.exe`, `BuildResultGame.exe` 실행 확인
- `Tools/MSBuild/msbuild.cmd`를 통한 `Release|x64 BuildResultGame` 직접 호출 성공
- `Bin/Release_x64`에 다음 파일 존재 확인:
  - `BuildResultGame.exe`
  - `KirbyGameDll.dll`
  - `libfbxsdk.dll`
  - `D3DX9_43.dll`
  - `Bitmaps/`
  - `FBX/`
  - `Map/`

즉 현재 기준으로는:

- VS Code 실행/디버그
- 수동 `Invoke-MSBuild.ps1`
- 에디터 내부 `Build Game`이 사용하는 MSBuild 경로

를 모두 함께 고려한 상태로 빌드/실행 설정을 정리한 것입니다.

### 19-7. 호환성 원칙 정리

이 프로젝트의 설정을 만질 때는 아래 원칙을 지키는 것이 좋습니다.

- `powershell`, `cmd`, `msbuild` 같은 **짧은 명령 이름만 믿지 말 것**
- 가능하면 Windows 기본 경로나 Visual Studio 표준 설치 경로도 함께 고려할 것
- `VS Code 빌드 흐름`과 `에디터 내부 Build Game 흐름`은 서로 다르다는 점을 항상 구분할 것
- `BuildOutput` 배포 품질은 exe/dll만이 아니라 `D3DX9_43.dll`, `libfbxsdk.dll`, `SceneData`, `Resources`까지 함께 확인할 것

요약하면 현재 설정의 목표는 아래 한 문장으로 정리할 수 있습니다.

**"기존 개발 PC에서는 그대로 동작하고, PATH/셸 환경이 다른 PC에서도 VS Code 실행과 BuildOutput 패키징이 최대한 같은 방식으로 성공하도록 빌드 진입 경로를 고정한다."**
