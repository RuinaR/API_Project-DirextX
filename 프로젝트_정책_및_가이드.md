# 프로젝트 정책 및 가이드

이 문서는 현재 `Solution_Kirby` 기준으로 엔진과 에디터가 따르는 주요 규칙을 한글로 정리한 안내서입니다.

목적은 두 가지입니다.

- 사용자가 확장할 때 어디까지가 지원되는 방식인지 빠르게 파악하기
- 지금 엔진이 왜 이런 정책을 택했는지 쉽게 이해하기

상세 이벤트 시점은 상위 폴더의 `GameObject_이벤트_호출시점.md`를 함께 참고하면 됩니다. 이 문서는 그보다 조금 더 넓게, 프로젝트 전체 정책을 요약합니다.

---

## 1. 기본 구조

현재 프로젝트는 크게 아래 구조를 따릅니다.

- `EngineFramework`
  - 공용 엔진 코드
  - 렌더링, 입력, 물리, 씬 저장/로드, 에디터 UI
- `KirbyGameDll`
  - 게임별 확장 코드
  - 사용자 `Component`, 사용자 `Action`, 테스트 씬용 코드
- `SceneData`
  - 씬 저장 데이터

기본 철학은 다음과 같습니다.

- 엔진 공통 기능은 `EngineFramework`에 둡니다.
- 게임별 규칙과 오브젝트 동작은 `KirbyGameDll`에 둡니다.
- 기존 구조를 한 번에 갈아엎기보다, 동작을 유지하면서 규칙을 점진적으로 정리합니다.

이 방향을 택한 이유:

- 엔진 코드와 게임 코드를 섞지 않기 위해서입니다.
- 나중에 다른 게임 DLL을 붙이거나 테스트 코드를 걷어내기 쉬워집니다.

## 2. 사용자 Component / Action 등록 방식

### 사용자 Component

사용자 컴포넌트는 아래 경로에 둡니다.

- `KirbyGameDll/UserComponents/Scripts`

등록은 아래 파일에서 합니다.

- `KirbyGameDll/UserComponents/GameComponentRegistry.cpp`

규칙:

- `Component`를 상속해서 구현합니다.
- `GetSerializableType()`를 구현합니다.
- 필요한 경우 `DrawInspector()`, `Serialize()`, `Deserialize()`를 구현합니다.
- 레지스트리에 등록해야 에디터 추가와 SceneData 복원이 됩니다.

이 정책을 택한 이유:

- 사용자 코드 위치를 고정해야 관리가 쉽습니다.
- SceneData 로드와 에디터 UI가 같은 등록 체계를 공유할 수 있습니다.

### 사용자 Action

사용자 액션은 아래 경로에 둡니다.

- `KirbyGameDll/UserActions/Scripts`

등록은 아래 파일에서 합니다.

- `KirbyGameDll/UserActions/GameActionRegistry.cpp`

규칙:

- `UIButton`에는 callback 본체가 저장되지 않습니다.
- SceneData에는 `actionKey` 문자열만 저장됩니다.
- 로드 시 `UIActionRegistry`에서 같은 key로 다시 바인딩됩니다.

이 정책을 택한 이유:

- lambda, 함수 포인터, 런타임 객체 포인터는 SceneData에 안전하게 저장할 수 없기 때문입니다.
- 문자열 key 방식이 저장/복원에 가장 단순하고 안정적입니다.

### 재빌드 정책

현재 C++ 사용자 코드 hot reload는 지원하지 않습니다.

규칙:

- 새 사용자 `Component` / `Action` 추가 후에는 DLL을 다시 빌드해야 합니다.
- 에디터도 다시 실행하는 것을 기본으로 봅니다.

이 정책을 택한 이유:

- 현재 구조에서 C++ 타입 등록과 DLL 로딩을 런타임에 완전히 안전하게 갱신하기 어렵기 때문입니다.

## 3. GameObject / Transform 정책

현재 비-UI 오브젝트는 `local transform`과 `world transform`을 분리해서 사용합니다.

핵심 개념:

- `localPosition`
- `localRotation`
- `localScale`
- `worldPosition`
- `worldRotation`
- `worldScale`

현재 보장되는 것:

- 부모 이동 시 자식도 같이 이동합니다.
- 부모 회전 시 자식이 부모 기준으로 같이 회전합니다.
- 부모 scale도 자식에 상속됩니다.
- `SetParent()` 시 월드 위치를 유지하면서 local 값이 다시 계산됩니다.

이 정책을 택한 이유:

- 예전의 단순 “위치만 따라가는 자식” 구조로는 계층 오브젝트를 제대로 표현하기 어려웠기 때문입니다.
- 물리 body, 렌더 위치, 에디터 이동 모두 같은 transform 의미를 공유해야 했기 때문입니다.

## 4. UI 오브젝트 구조

UI는 일반 월드 오브젝트와 다른 규칙으로 움직입니다.

`UIElement` 계열은 다음 값을 중심으로 배치됩니다.

- `Anchor Preset`
- `Local Offset`
- `Size`

여기서 중요한 점:

- `GameObject Position`은 UI에서 입력값이 아니라 계산 결과값입니다.
- 실제 배치 조절값은 `Local Offset`입니다.

즉 UI 배치 계산은 대략 이렇게 이해하면 됩니다.

- 기준점 선택: `Anchor Preset`
- 기준점으로부터 상대 이동: `Local Offset`
- 최종 위치 계산: `Position`

이 정책을 택한 이유:

- 반응형 UI에서는 절대 좌표 하나만으로는 창 크기 변화에 대응하기 어렵기 때문입니다.
- anchor + offset 구조가 부모-자식 UI에도 더 잘 맞습니다.

### 인스펙터 정책

`UIElement`가 붙은 오브젝트는 인스펙터에서:

- `GameObject Position`은 읽기 전용 결과값으로 보여줍니다.
- 실제 편집은 `UIElement`의 `Local Offset`에서 합니다.

이 정책을 택한 이유:

- UI에서 `Position`을 직접 수정하는 UX는 의미가 헷갈리기 때문입니다.
- 사용자가 실제로 만져야 하는 값은 `Local Offset`이기 때문입니다.

## 5. UI Anchor Preset 계산 방식

현재 지원하는 anchor preset 예시는 다음과 같습니다.

- `TopLeft`
- `TopCenter`
- `TopRight`
- `MiddleLeft`
- `MiddleCenter`
- `MiddleRight`
- `BottomLeft`
- `BottomCenter`
- `BottomRight`

### 루트 UI

부모가 없는 UI는 현재 UI 캔버스 크기를 기준으로 anchor를 계산합니다.

즉:

- 에디터 모드에서는 현재 에디터 화면 기준
- 게임 모드에서는 현재 게임 창 client 기준

으로 기준점이 정해집니다.

### 부모-자식 UI

부모가 있는 UI는 부모 `GameObject`가 아니라 **부모 `UIElement`의 rect**를 기준으로 anchor를 계산합니다.

즉 자식 UI는 다음 순서로 배치됩니다.

1. 부모 UI의 최종 위치와 크기를 구함
2. 부모 rect 안에서 anchor 기준점을 구함
3. 자식 자신의 anchor 기준점을 함께 계산함
4. `Local Offset`을 더해서 최종 위치를 만듦

간단히 말하면:

- 부모 기준점만 쓰는 것이 아니라
- 자식 자신의 anchor 기준점도 같이 반영합니다

예:

- 부모 버튼이 `TopLeft`
- 자식 라벨이 `MiddleCenter`

이면 자식은 부모 중앙 기준으로 배치됩니다.

이 정책을 택한 이유:

- `TopCenter`, `MiddleCenter`, `BottomRight` 같은 preset이 실제 의미대로 동작하게 하려면 자기 자신의 anchor 오프셋도 같이 계산해야 하기 때문입니다.

## 6. UI 계층 정책

현재 UI는 부모-자식 계층을 지원합니다.

권장 구조:

- 부모 `UIButton`
- 자식 `UILabel`
- 필요하면 그 아래 또 다른 `UILabel` / `UIImage`

### UI와 비-UI 부모자식 혼합 금지

현재는 다음 관계를 막고 있습니다.

- UI 오브젝트 밑에 비-UI 오브젝트
- 비-UI 오브젝트 밑에 UI 오브젝트

적용 경로:

- 코드에서 `SetParent(...)`
- 씬 로드 중 parent 복원
- 에디터 하이어라키 드래그 앤 드롭

이 정책을 택한 이유:

- UI는 화면 좌표 기준이고, 일반 오브젝트는 월드 좌표 기준이라 의미가 다르기 때문입니다.
- 둘을 같은 계층에 섞으면 transform, 입력, 카메라 영향이 금방 꼬입니다.

### UI 오브젝트 1개당 UIElement 1개만 허용

현재 같은 `GameObject`에는 `UIElement` 계열 컴포넌트를 하나만 허용합니다.

즉 막히는 경우:

- `UIButton + UILabel`을 같은 오브젝트에 직접 붙이는 것
- `UIImage + UILabel`을 같은 오브젝트에 직접 붙이는 것

대신 권장 방식:

- 부모/자식 `GameObject`로 나눠서 붙입니다.

이 정책을 택한 이유:

- 하나의 오브젝트에 UI rect 대표자가 여러 개 있으면 위치, 크기, hit test 기준이 모호해지기 때문입니다.

## 7. UI와 일반 Render 컴포넌트 혼합 정책

현재 같은 `GameObject`에는 아래 혼합을 금지합니다.

- `UIElement` 계열 + 일반 렌더 컴포넌트

일반 렌더 예:

- `ImageRender`
- `AnimationRender`
- `FBXRender`

적용 경로:

- 코드에서 `AddComponent(...)`
- 씬 로드 중 component 복원
- 인스펙터 Add Component 메뉴

즉 지금 규칙은 다음과 같습니다.

- UI 오브젝트는 UI 전용으로 쓴다
- 월드 오브젝트는 월드 렌더 전용으로 쓴다

이 정책을 택한 이유:

- UI는 UI projection과 UI 입력 체계를 쓰고
- 일반 렌더는 월드 transform과 카메라 투영을 쓰기 때문입니다.
- 두 계열을 같은 오브젝트에 섞으면 렌더, 선택, 입력 기준이 모호해집니다.

참고:

- `UIImage` 내부에서 엔진 구현용으로 `ImageRender`를 따로 쓰는 것은 사용자 공개 API 조합이 아니므로 예외로 봅니다.

## 8. 물리 객체 정책

### 기본 원칙

- `Rigidbody2D`가 있으면 `Collider2D`들은 그 body에 fixture로 붙습니다.
- `Rigidbody2D`가 없으면 단일 collider는 fallback body를 직접 만들 수 있습니다.
- 이때 fallback body는 현재 `static body`로 고정됩니다.

### 비권장 구조

- `Rigidbody2D` 없이 collider 여러 개를 같은 오브젝트에 붙이는 것

이 정책을 택한 이유:

- collider 여러 개가 body를 공유하지 않으면 물리 상태가 모호해지고 관리가 복잡해지기 때문입니다.
- 여러 collider를 함께 쓰고 싶으면 `Rigidbody2D`를 붙여 하나의 body를 공유하는 것이 가장 안정적입니다.
- `Rigidbody2D` 없이 collider만 둔 오브젝트는 “움직이는 물리 오브젝트”가 아니라 “정적 충돌체”로 보는 쪽이 현재 엔진 규칙과 가장 잘 맞습니다.

### Active와 물리의 관계

현재 `SetActive(false)`는 단순히 `Update()`만 멈추는 것이 아니라:

- physics body / fixture도 비활성화합니다
- 기존 collision/trigger pair는 정리하면서 `Exit` 이벤트를 전달할 수 있습니다

반대로 `SetActive(true)`는:

- 현재 `GameObject` transform을 다시 physics body에 동기화하고
- 그 뒤에 body를 활성화합니다

이 정책을 택한 이유:

- inactive 상태에서 충돌이 계속되는 것을 막기 위해서입니다.
- inactive 중 위치를 바꿨다가 다시 켤 때 물리 위치가 어긋나지 않게 하기 위해서입니다.

## 9. 충돌 / 트리거 이벤트 정책

### Collision 계열

- `CollisionEnter/Stay/Exit(Collider2D* other)`
- 둘 다 non-trigger일 때만 들어옵니다.

### Trigger 계열

- `TriggerEnter/Stay/Exit(Collider2D* other)`
- 둘 중 하나라도 trigger면 양쪽 모두에 들어옵니다.

### inactive / destroy 조건

`Enter` / `Stay` 계열은 다음이면 전달하지 않습니다.

- receiver가 inactive
- other가 inactive
- receiver가 destroy pending
- other가 destroy pending

`Exit`는 종료 알림 성격이라 조금 다릅니다.

- 상대가 이미 사라졌으면 `nullptr`로 들어올 수 있습니다.

즉 아래 상황이 가능합니다.

- `OnCollisionExit(nullptr)`
- `OnTriggerExit(nullptr)`

이 정책을 택한 이유:

- 종료 알림은 마지막 정리 기회이기 때문입니다.
- 상대가 이미 제거되었더라도 “끝났다”는 사실은 알려줄 필요가 있습니다.

## 10. Enable / Disable 이벤트 정책

현재 `OnEnable()`과 `OnDisable()`는 유니티의 전체 생명주기 의미와는 조금 다릅니다.

### `OnEnable()`

현재 의미:

- `SetActive(false -> true)` 전환 알림

호출되지 않는 경우:

- 객체를 처음 생성했을 때
- SceneData에서 active 상태로 처음 로드됐을 때
- 이미 active인데 `SetActive(true)`를 다시 호출했을 때

### `OnDisable()`

현재 의미:

- `SetActive(true -> false)` 전환 알림

이 정책을 택한 이유:

- 지금 엔진은 생성 이벤트와 active 전환 이벤트를 분리해서 단순하게 관리하기 위해서입니다.
- 초기 생성 훅까지 섞으면 호출 시점이 더 복잡해집니다.

---

## 11. 마우스 / Hover / 버튼 이벤트 정책

지원되는 대표 이벤트:

- `OnLBtnDown()`
- `OnLBtnUp()`
- `OnRBtnDown()`
- `OnRBtnUp()`
- `OnMouseHoverEnter()`
- `OnMouseHoverStay()`
- `OnMouseHoverExit()`

기본 규칙:

- 현재 마우스가 가리키는 활성 오브젝트에만 들어갑니다.
- destroy 상태 오브젝트에는 들어가지 않습니다.

UI와 월드 오브젝트 모두 같은 `GameObject -> Component` fan-out 구조로 전달됩니다.

이 정책을 택한 이유:

- 입력 처리 경로를 하나로 유지해야 디버깅이 쉽기 때문입니다.

---

## 12. 리사이즈 / 렌더 경로 정책

현재 에디터와 게임은 최대한 같은 렌더 철학을 공유합니다.

정리된 방향:

- 에디터와 게임 모두 현재 창 크기를 기준으로 렌더합니다.
- 게임모드도 별도 고정 letterbox viewport를 강제하지 않습니다.
- 카메라 projection은 현재 창 비율을 반영합니다.
- UI 루트 anchor도 현재 UI 캔버스 크기를 기준으로 계산합니다.

이 정책을 택한 이유:

- 에디터에서는 맞는데 게임에서는 안 맞는 문제를 줄이기 위해서입니다.
- UI 계층과 리사이즈 동작을 같은 기준으로 검증하기 위해서입니다.

주의:

- UI는 반응형 anchor를 지원하지만, 모든 레이아웃이 자동으로 “예쁘게” 재배치되는 것은 아닙니다.
- anchor와 offset을 어떤 기준으로 잡았는지가 여전히 중요합니다.

## 13. SceneData 관련 메모

UI anchor 구조가 들어가면서 UI 저장 데이터는 anchor preset과 local offset을 같이 사용합니다.

현재 원칙:

- 런타임 포인터는 저장하지 않습니다.
- 문자열 key, 숫자 값, bool 값처럼 안전한 데이터만 저장합니다.
- UI는 `anchorPreset`, `position(local offset)`, `size`를 중심으로 저장합니다.

이 정책을 택한 이유:

- 씬 저장/로드를 안정적으로 유지하기 위해서입니다.
- DirectX 포인터, Box2D body 포인터, callback 본체는 저장 대상이 아니기 때문입니다.

---

## 14. 사용 팁

### UI를 만들 때

- 한 오브젝트에 UIElement는 하나만 붙입니다.
- 여러 UI를 조합할 때는 부모-자식 `GameObject`로 나눕니다.
- 위치 조절은 `GameObject Position`보다 `Local Offset`으로 합니다.
- 부모 안에서 정렬하고 싶으면 자식의 `Anchor Preset`을 적극적으로 씁니다.

### 물리 오브젝트를 만들 때

- collider가 여러 개면 `Rigidbody2D`를 같이 붙이는 쪽을 권장합니다.
- inactive 상태에서 물리를 끄고 싶으면 `SetActive(false)`를 사용합니다.

### 사용자 확장을 만들 때

- `EngineFramework`가 아니라 `KirbyGameDll`에 둡니다.
- registry 등록을 빼먹지 않습니다.
- SceneData에 저장될 값과 저장되면 안 되는 값을 구분합니다.

## 15. 한 줄 요약

현재 프로젝트 정책은 아래 한 줄로 요약할 수 있습니다.

- 월드 오브젝트와 UI 오브젝트의 책임을 분리하고, 저장/복원과 입력/렌더 규칙을 단순하고 예측 가능하게 유지한다.

이 방향을 유지하면 에디터와 런타임 차이를 줄이면서도, 사용자 확장 코드를 비교적 안전하게 쌓아갈 수 있습니다.
