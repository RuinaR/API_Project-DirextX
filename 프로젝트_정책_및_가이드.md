# 프로젝트 정책 및 가이드

이 문서는 현재 `Solution_Kirby` 기준으로 엔진, 에디터, 사용자 확장 코드 전반에 적용하는 주요 규칙을 정리한 문서입니다.

정책을 택한 이유나 문제 해결 과정은 `설계_배경_및_문제해결_기록.md`로 분리했습니다. 이 문서는 현재 기준만 빠르게 확인하는 용도입니다.

## 1. 큰 방향

현재 프로젝트는 기존 프레임워크를 점진적으로 "미니 게임엔진 + 에디터"에 가깝게 정리하는 단계입니다.

장기 목표:

- `EngineRuntime`
  - 게임 실행에 필요한 기능만 포함
- `EngineEditor`
  - Hierarchy, Inspector, BuildSettings, Scene 편집, ImGui 기반 도구 포함
- `KirbyGameDll`
  - 게임별 `Component`, `Action`만 포함
  - 장기적으로 `EngineRuntime`만 의존
- `EditorApp`
  - `EngineRuntime + EngineEditor + KirbyGameDll`
- `GameApp`
  - `EngineRuntime + KirbyGameDll`

현재는 아직 완전히 나뉘지 않았고, `EngineFramework` 안에 Runtime과 Editor 책임이 함께 들어 있습니다.

## 2. 기본 구조 정책

- 엔진 공통 기능은 `EngineFramework`
- 게임별 확장 기능은 `KirbyGameDll`
- Scene 저장 데이터는 `SceneData/*.json`
- 에디터 실행기는 `BuildEdit`
- 게임 실행기는 `BuildResultGame`

## 3. 사용자 확장 정책

### User Component

- 위치:
  - `KirbyGameDll/UserComponents/Scripts`
- 등록:
  - `KirbyGameDll/UserComponents/GameComponentRegistry.cpp`

규칙:

- `Component`를 상속합니다.
- `GetSerializableType()`을 구현합니다.
- 필요하면 `DrawInspector()`, `Serialize()`, `Deserialize()`, `ResolveReferences()`를 구현합니다.

### User Action

- 위치:
  - `KirbyGameDll/UserActions/Scripts`
- 등록:
  - `KirbyGameDll/UserActions/GameActionRegistry.cpp`

규칙:

- `UIButton`은 callback 본체를 저장하지 않습니다.
- SceneData에는 `actionKey`만 저장합니다.
- 로드 시 registry에서 다시 바인딩합니다.

### hot reload 정책

- 현재 사용자 C++ `Component/Action` hot reload는 지원하지 않습니다.
- 새 C++ 확장을 추가하거나 수정하면 `KirbyGameDll.dll`을 다시 빌드하고 에디터를 재실행하는 것을 기본으로 합니다.

## 4. GameObject / 계층 정책

- `GameObject`는 local/world transform hierarchy를 가집니다.
- `SetParent()` 시 world transform은 유지하고 local transform을 재계산합니다.
- parent-child 관계는 SceneData에서 `parentId` 기준으로 복원합니다.

금지 정책:

- 자기 자신을 부모로 설정 금지
- 자식 계층을 부모로 설정 금지
- UI 오브젝트와 비UI 오브젝트의 부모-자식 혼합 금지

## 5. UI 정책

- `UIElement` 계열은 UI 전용 오브젝트로 취급합니다.
- UI는 `anchor preset + local offset + size` 중심으로 배치합니다.
- inspector에서 UI 위치를 직접 만질 때는 `GameObject Position`보다 `UIElement` 쪽 배치 값을 기준으로 봅니다.

금지 정책:

- 같은 `GameObject`에 `UIElement` 계열을 여러 개 직접 붙이지 않음
- 같은 `GameObject`에 `UIElement` 계열과 일반 월드 렌더 컴포넌트를 함께 붙이지 않음

## 6. 물리 정책

- 물리는 현재 2D 전용 기준입니다.
- 주요 타입:
  - `Rigidbody2D`
  - `Collider2D`
  - `BoxCollider2D`
  - `CircleCollider2D`
  - `Physics2D`

규칙:

- `Rigidbody2D`가 있으면 collider는 그 body의 fixture로 붙습니다.
- `Rigidbody2D`가 없으면 collider가 static fallback body를 직접 생성합니다.
- `Rigidbody2D` 없이 collider 여러 개를 한 오브젝트에 붙이는 fallback 구조는 비권장입니다.

## 7. 충돌 / 트리거 이벤트 정책

- 실제 충돌 이벤트는 Box2D `ContactListener` 기반입니다.
- `fixture userData`에서 `Collider2D*`를 읽습니다.
- 둘 다 non-trigger일 때만 `CollisionEnter/Stay/Exit`
- 둘 중 하나라도 trigger면 `TriggerEnter/Stay/Exit`

추가 정책:

- inactive 또는 destroy pending 대상에는 `Enter/Stay`를 보내지 않습니다.
- 삭제 예약 또는 강제 종료 상황에서는 `OnCollisionExit(nullptr)`, `OnTriggerExit(nullptr)`를 허용합니다.
- `DestroyObject()`는 즉시 delete가 아니라 pending remove입니다.

## 8. SceneData 정책

### 저장하는 것

- scene 이름
- timeScale
- camera position / rotation / projection
- object id / parentId / name / tag / active / transform
- component type 문자열
- component별 직렬화 데이터
- asset key / path 문자열
- `UIButton`의 `actionKey`
- object reference id

### 저장하지 않는 것

- raw pointer
- `IDirect3DTexture9*`
- `b2Body*`, fixture pointer
- callback / lambda 본체
- DirectX runtime object
- Box2D runtime object
- editor UI 상태

### 버전 정책

- 최신 저장 버전은 현재 `6`
- v3 구버전 SceneData도 계속 로드 가능
- `BoxCollider2D`가 canonical type
- legacy `"BoxCollider"` alias는 호환용으로 유지

## 9. 테스트 / 샘플 정책

- 테스트용 컴포넌트와 실제 게임 로직 컴포넌트는 구분합니다.
- 테스트 씬과 실제 게임 씬도 문서 기준으로 구분합니다.

현재 샘플 유지:

- `SampleSpinComponent`
- `Physics2DKeyboardTestComponent`
- `CollisionVisualTestComponent`

현재 임시 테스트 유지:

- `ActivePhysicsTestStatusComponent`
- `ActivePhysicsTestInfoComponent`

현재 대표 테스트 씬:

- `SceneData/Sample.json`
- `SceneData/Physics2DTestScene.json`
- `SceneData/Physics2DInteractionTestScene.json`
- `SceneData/ActivePhysicsToggleTestScene.json`

## 10. Runtime / Editor 경계 정책

현재는 아직 완전히 분리되지 않았지만, 아래 방향을 기준으로 구조를 정리합니다.

### Runtime에 남겨야 할 책임

- frame loop
- device 생성/해제
- physics world와 collision dispatch
- object lifecycle
- 공통 scene render
- SceneData save/load core
- input / raycast / camera / projection

### Editor로 이동해야 할 책임

- ImGui bootstrap과 editor font 정책
- hierarchy / inspector / resource browser
- scene dirty / snapshot / open / save-as workflow
- editor playback controller
- selected object marker
- editor 전용 hover / mouse capture 정책

## 11. 현재 주의할 구조적 혼합 지점

- `MainFrame`
  - ImGui 초기화
  - editor playback
- `RenderManager`
  - `EditUpdate()`
  - selected marker
  - editor frame 창
- `ObjectManager`
  - `ImguiUpdate()`
  - editor window 직접 호출
- `SceneDataManager`
  - dirty / snapshot / open / new / save-as
- `WindowFrame`
  - scene open 정책과 오류 UI 처리

## 12. 작업 원칙

- 새 기능을 넣을 때는 가능한 Runtime 책임과 Editor 책임을 의식하고 분리 가능한 방향으로 작성합니다.
- SceneData 포맷은 필요 없는 변경을 피합니다.
- 테스트/샘플 코드라고 해도 현재 실제 씬에서 사용 중이면 성급히 제거하지 않습니다.
- deprecated 파일은 실제 활성 사용처와 프로젝트 항목 상태를 먼저 확인한 뒤 정리합니다.
