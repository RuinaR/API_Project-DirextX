# 프로젝트 정책 및 가이드

이 문서는 현재 `KirbyEngine`에서 실제로 적용하고 있는 규칙을 빠르게 정리한 문서입니다.

왜 이런 규칙을 택했는지 자세한 이유는 `설계_배경_및_문제해결_기록.md`를 보면 됩니다.

## 1. 큰 방향

현재 프로젝트의 큰 방향은 아래와 같습니다.

- 엔진 공통 기능은 `EngineFrameworkDll`
- editor 전용 기능은 `EngineEditor` 쪽으로 점진적으로 분리
- 게임별 확장 코드는 `KirbyGameDll`
- 에디터 실행기는 `EditorApp`
- 게임 실행기는 `GameApp`

아직 완전히 분리된 최종 상태는 아니고,
runtime/editor 책임이 일부 섞여 있는 부분을 계속 정리 중입니다.

## 2. 사용자 코드 위치

### 사용자 컴포넌트

- 위치: `KirbyGameDll/UserComponents/Scripts`
- 등록: `KirbyGameDll/UserComponents/GameComponentRegistry.cpp`

### 사용자 액션

- 위치: `KirbyGameDll/UserActions/Scripts`
- 등록: `KirbyGameDll/UserActions/GameActionRegistry.cpp`

## 3. 사용자 컴포넌트 기본 규칙

사용자 컴포넌트는 보통 아래를 구현합니다.

- `GetSerializableType()`
- 필요하면 `DrawInspector()`
- 필요하면 `Serialize()` / `Deserialize()`
- 필요하면 `ResolveReferences()`
- registry 기반 참조 필드를 쓸 경우 `RegisterReferenceFields()`

## 4. GameObject / 계층 규칙

- `GameObject`는 parent-child 구조를 가질 수 있습니다.
- `SetParent()` 시 world transform은 유지하고 local transform을 다시 계산합니다.
- SceneData에서는 `parentId`를 기준으로 계층을 복원합니다.

### 이름과 태그

- `name`
  - 에디터에서 보는 이름
  - scene 안에서 중복되지 않게 관리하는 쪽을 기본으로 둠
- `tag`
  - gameplay 분류용 문자열
  - 중복 허용
  - 기본값은 `Untagged`

### 금지 규칙

- 자기 자신을 부모로 설정 금지
- 자식 계층을 부모로 설정 금지
- UI 오브젝트와 비UI 오브젝트를 부모-자식으로 섞지 않음

## 5. UI 관련 규칙

- `UIElement` 계열은 UI 전용 오브젝트로 봅니다.
- 일반 월드 렌더 컴포넌트와 UI 전용 렌더 컴포넌트를 같은 `GameObject`에 같이 두지 않습니다.
- UI 배치는 `anchor + local offset + size` 기준으로 보는 편이 맞습니다.

## 6. Physics2D 규칙

- 현재 물리는 2D 전용입니다.
- 주요 타입:
  - `Rigidbody2D`
  - `Collider2D`
  - `BoxCollider2D`
  - `CircleCollider2D`

### 사용 규칙

- `Rigidbody2D`가 있으면 collider는 그 body에 붙습니다.
- `Rigidbody2D`가 없으면 collider가 static fallback body를 만듭니다.
- collider 여러 개가 필요한 경우 가능하면 `Rigidbody2D`를 같이 쓰는 쪽을 권장합니다.

## 7. 충돌 / 트리거 규칙

- 둘 다 일반 collider면 collision 이벤트 사용
- 둘 중 하나라도 trigger면 trigger 이벤트 사용

### 추가 규칙

- inactive 또는 destroy pending 대상에는 `Enter/Stay`를 보내지 않습니다.
- `Exit` 이벤트에서는 `nullptr`를 허용합니다.
  - `OnCollisionExit(nullptr)`
  - `OnTriggerExit(nullptr)`
- `DestroyObject()`는 즉시 삭제가 아니라 예약 삭제 방식입니다.

## 8. SceneData 규칙

### 저장하는 것

- scene 이름
- timeScale
- camera 정보
- object id / parentId / name / tag / active / transform
- component type
- component data
- `componentId`
- registry 기반 `referenceFields`
- `UIButton`의 `actionKey`

### 저장하지 않는 것

- raw pointer
- DirectX runtime object
- Box2D runtime object
- callback / lambda 본체
- editor UI 상태

### 현재 버전

- 최신 SceneData 버전: `9`

### 호환 규칙

- 구버전 SceneData도 가능한 범위에서 계속 읽습니다.
- 기존 key는 함부로 지우지 않습니다.
- 저장 포맷을 바꿀 때는 가능하면 version bump로 관리합니다.

## 9. ReferenceFieldRegistry 규칙

registry 기반 참조 필드는 아래 규칙을 따릅니다.

- 자동 reflection 시스템은 만들지 않음
- `RegisterReferenceFields(...)`에 등록된 필드만 공통 처리
- `GameObject` 참조와 `Component` 참조를 구분해서 관리
- 저장 시 label 기준으로 기록
- 로드 시 label 기준으로 다시 매칭

### 현재 지원

- registry 기반 inspector 표시
- hierarchy에서 GameObject drag/drop
- componentId 기반 component reference 저장/로드

## 10. plugin / 게임 DLL 규칙

현재 plugin 표면은 아래만 유지합니다.

- `RegisterGameContent(HINSTANCE, const AppLaunchDesc&)`
- `GetName()`
- `ReleaseGameContent()`

즉, 지금 `KirbyGameDll`은
"게임별 컴포넌트/액션 등록과 정리" 역할에 집중하는 방향입니다.

## 11. editor/runtime 분리 방향

장기 방향은 아래와 같습니다.

- `EngineFrameworkDll`
  - runtime core 중심
- `EngineEditor`
  - editor UI / workflow 중심
- `KirbyGameDll`
  - 게임별 확장 코드

현재는 아직 아래 파일들에 editor/runtime 책임이 섞여 있습니다.

- `MainFrame`
- `WindowFrame`
- `RenderManager`

그래서 새 기능을 넣을 때도,
가능하면 "runtime 쪽인지 editor 쪽인지"를 먼저 생각하고 작성하는 것을 기준으로 삼습니다.

## 12. 테스트 / 샘플 정책

- 테스트 컴포넌트와 실제 게임 로직은 구분해서 본다.
- 샘플 씬과 검증 씬도 역할을 구분해서 유지한다.
- 지금 실제로 씬에서 쓰고 있는 테스트/샘플 코드는 성급히 삭제하지 않는다.

## 13. 작업할 때 기본 원칙

- 동작하는 기능을 먼저 유지한다.
- SceneData 호환을 쉽게 깨지 않게 한다.
- 구조 정리는 한 번에 크게 갈아엎기보다 단계적으로 진행한다.
- include 경계와 DLL 경계를 같이 본다.
- editor 기능이 runtime에 섞여 있으면 기록부터 남기고, 그 다음 분리한다.

## TimeScale 사용 기준

현재 프로젝트에서는 사용자 컴포넌트가 시간을 사용할 때 아래 기준을 기본으로 둡니다.

- 기본 게임 로직은 `DeltaTime()` 기준으로 구현한다.
- `DeltaTime()`은 `TimeScale` 영향을 받는다.
- 타임스케일 영향을 받지 않아야 하는 특별한 연출만 `UnscaledDeltaTime()`을 사용한다.

예:

- 이동, 점프, 일반 타이머, 패턴 진행: `DeltaTime()`
- 잠깐 멈춤 연출 중에도 계속 돌아야 하는 회전/효과: `UnscaledDeltaTime()`
