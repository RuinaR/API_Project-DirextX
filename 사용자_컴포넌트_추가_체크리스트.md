# 사용자 컴포넌트 추가 체크리스트

이 문서는 `KirbyGameDll`에 새 사용자 컴포넌트를 추가할 때 꼭 확인할 항목만 짧게 정리한 문서입니다.

## 1. 파일 만들기

새 컴포넌트 파일은 아래 폴더에 만듭니다.

- `Solution_Kirby/KirbyGameDll/UserComponents/Scripts/`

보통 2개 파일을 같이 만듭니다.

- `MyComponent.h`
- `MyComponent.cpp`

## 2. 기본 클래스 상속

새 컴포넌트는 `Component`를 상속해서 만듭니다.

예시:

```cpp
class MyComponent : public Component
{
public:
    static constexpr const char* kComponentType = "MyComponent";

    void Initialize() override;
    void Release() override;
    void Start() override;
    void Update() override;

    const char* GetInspectorName() const override;
    void DrawInspector() override;
    const char* GetSerializableType() const override;
    std::string Serialize() const override;
    bool Deserialize(const std::string& componentJson) override;
};
```

## 3. 이름 맞추기

아래 값은 서로 맞춰 두는 편이 좋습니다.

- `kComponentType`
- `GetSerializableType()` 반환값
- `GameComponentRegistry.cpp`에 등록하는 type 이름

이 값이 다르면 저장/로드할 때 헷갈릴 수 있습니다.

## 4. vcxproj에 파일 추가

VSCode에서 파일만 만들면 빌드 대상에 자동으로 들어가지 않을 수 있습니다.

그래서 아래 파일도 같이 수정해야 합니다.

- `KirbyGameDll/KirbyGameDll.vcxproj`
- 필요하면 `KirbyGameDll/KirbyGameDll.vcxproj.filters`

즉, 폴더에 파일을 넣는 것만으로 끝나지 않습니다.

## 5. 컴포넌트 등록

새 컴포넌트는 아래 파일에서 등록합니다.

- `KirbyGameDll/UserComponents/GameComponentRegistry.cpp`

예시:

```cpp
REGISTER_GAME_COMPONENT(factory, MyComponent, "My Component", "User");
```

## 6. Inspector 만들기

에디터에서 값을 바꾸고 싶으면 `DrawInspector()`를 구현합니다.

예시:

```cpp
void MyComponent::DrawInspector()
{
    ImGui::Checkbox("Enabled", &m_enabled);
    ImGui::DragFloat("Speed", &m_speed, 0.1f);
}
```

## 7. 저장할 값 정리

씬 저장이 필요하면 아래 함수를 같이 구현합니다.

- `Serialize()`
- `Deserialize()`

중요한 점:

- 다시 계산 가능한 값은 굳이 저장하지 않아도 됩니다.
- 실제로 다음 실행 때 다시 필요한 값만 저장하는 편이 좋습니다.

## 8. 참조 필드가 있으면

`GameObject*`, `Component*` 같은 참조를 Inspector에 노출하고 싶으면
`RegisterReferenceFields(...)`를 같이 구현합니다.

예시:

```cpp
Component* m_targetRigidbody = nullptr;

void MyComponent::RegisterReferenceFields(ReferenceFieldRegistry& registry)
{
    registry.RegisterGameObjectRef("Target Object", &m_targetObject);
    registry.RegisterComponentRef("Target Rigidbody", &m_targetRigidbody, "Rigidbody2D");
}
```

## 9. 빌드 후 확인

새 C++ 컴포넌트를 추가했다면 보통 아래 순서로 확인합니다.

1. `KirbyGameDll` 빌드
2. `EditorApp` 실행
3. Add Component 메뉴에 보이는지 확인
4. Inspector가 정상 동작하는지 확인
5. 저장/로드가 되면 SceneData에도 반영되는지 확인

## 10. 자주 놓치는 부분

- 파일만 만들고 `KirbyGameDll.vcxproj`에 안 넣는 경우
- `kComponentType`와 `GetSerializableType()` 이름이 다른 경우
- `GameComponentRegistry.cpp` 등록을 빼먹는 경우
- Inspector 값은 바뀌는데 `Serialize()/Deserialize()`를 안 만들어서 저장이 안 되는 경우

## 한 줄 정리

사용자 컴포넌트 추가는
`파일 생성 -> vcxproj 추가 -> registry 등록 -> inspector/serialize 구현`
순서로 생각하면 가장 덜 헷갈립니다.

## 11. 시간값을 먼저 체크할 것
사용자 컴포넌트에서 시간 관련 로직을 넣을 때는 먼저 아래를 정합니다.

- 이 로직이 `TimeScale` 영향을 받아야 하는지
- 아니면 `TimeScale`과 무관하게 계속 돌아야 하는지

기본 기준은 아래처럼 보면 됩니다.

- 일반 로직: `DeltaTime()`
- 예외 연출: `UnscaledDeltaTime()`

예를 들어:

- 이동, 점프, 스크롤 같은 일반 플레이 로직은 `DeltaTime()`
- 잠깐 멈춤 연출 중에도 계속 돌아야 하는 회전이나 UI 효과는 `UnscaledDeltaTime()`

처음에는 `DeltaTime()`으로 만들고,
정말로 타임스케일 영향을 받지 않아야 할 때만 `UnscaledDeltaTime()`으로 바꾸는 편이 안전합니다.

## 12. Component reference field를 붙일 때

`RegisterReferenceFields(...)`에서 특정 컴포넌트를 Inspector로 받고 싶다면,
현재 구조에서는 아래 순서를 따르는 편이 안전합니다.

- Inspector에서 받을 때의 타입 제한은 `expectedType` 문자열로 건다.
- 실제 멤버 필드는 `Component*`로 둔다.
- 사용할 때만 `dynamic_cast`로 원하는 정확한 타입으로 꺼낸다.

예시:

```cpp
Component* m_targetRigidbody = nullptr;

void MyComponent::RegisterReferenceFields(ReferenceFieldRegistry& registry)
{
    registry.RegisterComponentRef(
        "Target Rigidbody",
        &m_targetRigidbody,
        "Rigidbody2D");
}
```

사용할 때:

```cpp
Rigidbody2D* rigidbody = dynamic_cast<Rigidbody2D*>(m_targetRigidbody);
```

왜 이렇게 하느냐면,
현재 `RegisterComponentRef(...)`가 `Component**`를 받기 때문입니다.
그래서 `Rigidbody2D**`나 `SpawnObjectPoolComponent**`를 그대로 넘기는 방식은 현재 구조와 바로 맞지 않습니다.
