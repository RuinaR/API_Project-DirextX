# 설계 배경 및 문제 해결 기록

이 문서는 현재 정책을 왜 택했는지, 어떤 문제를 실제로 겪으면서 지금 구조로 정리됐는지를 모아 둔 기록 문서입니다.

정책 자체의 최신 기준은 `README.md`, `프로젝트_정책_및_가이드.md`를 참고하면 됩니다.

## 1. 왜 한 번에 새 엔진으로 갈아엎지 않는가

현재 프로젝트는 이미 돌아가던 프레임워크 위에 기능을 얹는 방식으로 확장되어 왔습니다. 그래서 구조적으로 완벽하지 않은 지점이 있어도, 당장 전체를 새로 짜기보다는 동작을 유지하면서 점진적으로 정리하는 쪽을 택했습니다.

이 방향을 유지하는 이유는 다음과 같습니다.

- 이미 동작하는 Scene, Render, UI, Physics 경로를 잃지 않기 위해
- SceneData와 사용자 확장 코드를 계속 살린 채 리팩터링하기 위해
- “구조는 깨끗하지만 기능이 다시 안 되는 상태”를 피하기 위해

즉, 현재 프로젝트는 “완성된 엔진”보다 “동작을 유지하며 엔진화하는 과정”으로 보는 편이 맞습니다.

## 2. 왜 Runtime / Editor 분리를 장기 목표로 두는가

현재 `EngineFramework`에는 Runtime과 Editor 책임이 함께 있습니다. 실제로 점검해 보면 아래 같은 혼합이 존재합니다.

- `MainFrame`
  - DX9 device / physics step 같은 runtime 책임
  - ImGui 초기화 / editor playback 같은 editor 책임
- `RenderManager`
  - 공통 scene render
  - `EditUpdate()`, selected object marker, frame 창
- `ObjectManager`
  - object lifecycle
  - editor window 직접 호출
- `SceneDataManager`
  - save/load core
  - dirty/snapshot/open/save-as workflow

이 혼합은 작은 프로젝트 단계에서는 빠르게 기능을 넣는 데 유리했지만, 앞으로는 아래 문제가 커집니다.

- GameApp에서 editor 코드를 같이 끌고 가게 됨
- game runtime bug와 editor tool bug가 같은 계층에 얽힘
- 테스트와 책임 분리가 어려워짐

그래서 장기적으로는 `EngineRuntime`, `EngineEditor`, `KirbyGameDll`을 분리하는 방향을 기준으로 두고 있습니다.

## 3. SceneData를 JSON + type/data 구조로 정리한 이유

기존에도 저장/로드가 있었지만, component가 늘어나고 hierarchy, UI, Physics2D가 붙으면서 “무엇을 저장하고 무엇을 저장하지 않을지” 기준이 더 중요해졌습니다.

지금 구조가 `{ "type": "...", "data": { ... } }` 형태인 이유는 다음과 같습니다.

- component 추가 시 SceneData 구조를 통째로 바꾸지 않아도 되기 때문
- `ComponentFactory` 기반으로 type 문자열만 맞으면 복원이 가능하기 때문
- 런타임 포인터 대신 직렬화 가능한 데이터만 저장하기 쉬워지기 때문

특히 아래는 의도적으로 저장하지 않도록 정리했습니다.

- raw pointer
- `IDirect3DTexture9*`
- `b2Body*`
- callback / lambda 본체

이걸 명확히 하지 않으면 저장은 되어도 복원이 불가능하거나, editor와 game에서 서로 다른 상태를 만들기 쉬웠습니다.

## 4. SceneData 버전을 유지하면서 호환을 남겨 둔 이유

SceneData 구조를 손볼 때마다 과거 파일을 다 버리면 빠르긴 하지만, 실제로는 테스트 씬, 샘플 씬, 예전 작업본이 한 번에 깨집니다.

그래서 현재는 아래처럼 운영하고 있습니다.

- 최신 저장 버전은 `7`
- v3 같은 구버전 SceneData도 계속 로드 가능
- v7부터 `name/tag`를 실제 분리 저장
- v6 이하 SceneData는 기존 fallback으로 계속 로드 가능
- `BoxCollider2D`가 canonical type
- legacy `"BoxCollider"` alias는 계속 로드 가능

특히 `"BoxCollider"` alias를 바로 없애지 않은 이유는, C++ 코드 rename이 끝났더라도 예전 SceneData가 남아 있을 가능성을 초기에 배제할 수 없었기 때문입니다. 실제로 이번 정리 과정에서 원본 `SceneData/`와 `BuildOutput/SceneData/`를 구분해서 확인하는 절차도 따로 거쳤습니다.

## 4-1. 왜 GameObject의 name/tag를 분리했는가

처음 구조에서는 `tag`가 사실상 오브젝트 표시 이름 역할까지 같이 맡고 있었고, SceneData에서도 `name`과 `tag`에 같은 값을 저장하고 있었습니다.

이 상태는 작게는 inspector/hierarchy 의미가 섞이고, 크게는 검색 API와 gameplay 분류 정책이 뒤엉키는 문제를 만들었습니다.

그래서 현재는 아래처럼 역할을 분리했습니다.

- `name`
  - 사람이 읽는 오브젝트 이름
  - hierarchy / inspector / reference field 표시 기준
  - scene 내 중복 비허용 정책
- `tag`
  - gameplay grouping/filtering 문자열
  - 중복 허용
  - 기본값은 `Untagged`

이 분리를 안정적으로 반영하기 위해 저장 버전도 `7`로 올렸고, 기존 v4/v6 SceneData는 fallback으로 계속 읽히게 유지했습니다.

## 5. 왜 User Action은 callback이 아니라 actionKey만 저장하는가

버튼이나 UI 액션은 처음에는 “클릭하면 바로 함수 호출” 구조로 가기 쉽지만, SceneData 저장/로드가 붙는 순간 callback 본체는 저장 대상이 될 수 없습니다.

그래서 현재는 아래 정책을 유지합니다.

- SceneData에는 `actionKey`만 저장
- 로드 시 `UIActionRegistry`에서 다시 바인딩

이 정책은 조금 우회적으로 보이지만, 아래 문제를 피하는 데 효과적이었습니다.

- lambda 본체 저장 불가
- DLL 재로딩 후 함수 포인터 무효화
- editor에서 로드한 버튼이 예전 callback을 계속 잡고 있는 문제

즉, 저장 가능한 건 문자열 key뿐이고, 실제 동작은 실행 시 registry에서 다시 연결하는 쪽이 가장 안전했습니다.

## 6. 왜 사용자 확장 코드를 KirbyGameDll로 몰았는가

예전 구조처럼 테스트 코드나 게임별 코드를 `EngineFramework`에 바로 넣으면 당장은 편하지만, 시간이 지나면 엔진 코드와 게임 코드가 분리되지 않습니다.

지금 `KirbyGameDll`을 확장 계층으로 두는 이유는 다음과 같습니다.

- 엔진 공통 코드와 게임별 코드의 경계를 유지하기 위해
- SceneData, registry, editor inspector가 “게임별 확장도 공식 경로로 지원한다”는 구조를 만들기 위해
- 나중에 다른 게임 DLL이나 샘플 DLL을 붙일 여지를 남기기 위해

현재 hot reload는 지원하지 않지만, 대신 “등록 위치가 명확한 구조”를 먼저 정리하는 쪽을 우선했습니다.

## 7. 왜 UI와 월드 오브젝트 혼합을 금지하는가

UI와 월드 오브젝트는 둘 다 `GameObject` 기반으로 보이지만, 실제 좌표계와 렌더 기준은 다릅니다.

- 월드 오브젝트:
  - camera, projection, world transform 기준
- UI 오브젝트:
  - UI canvas, anchor, local offset 기준

이 둘을 같은 parent-child 계층에 섞거나, 같은 `GameObject`에 UI와 월드 렌더를 같이 붙이면 아래 같은 문제가 생겼습니다.

- 위치 기준이 모호해짐
- 입력 판정이 흔들림
- inspector에서 어느 값을 편집하는지 혼란스러움
- resize 이후 UI 반응형과 월드 transform이 충돌

그래서 지금은 “UI/비UI parent-child 금지”, “UIElement와 일반 렌더 공존 금지”를 명시 정책으로 두고 있습니다.

## 8. 왜 Physics2D fallback 구조를 허용하지만 비권장으로 두는가

현재 `Rigidbody2D`가 없으면 collider가 static fallback body를 직접 만듭니다. 이 구조는 단일 collider-only 오브젝트를 빠르게 만들 때는 편합니다.

하지만 collider를 여러 개 붙이는 순간 문제가 생깁니다.

- collider마다 body가 따로 생길 수 있음
- transform sync 기준이 오브젝트 단위가 아니라 collider 단위로 흩어질 수 있음
- 나중에 rigidbody를 붙였다 떼는 경로가 복잡해짐

그래서 현재 결론은 이렇습니다.

- fallback 자체는 허용
- 단일 collider-only 오브젝트는 지원
- 다중 collider fallback 구조는 비권장
- 다중 collider가 필요하면 `Rigidbody2D`를 붙여 하나의 body를 공유

즉, 사용성 때문에 완전히 막지는 않되, 장기적으로는 rigidbody 기반 공유 구조를 기본으로 보는 정책입니다.

## 9. 왜 collision / trigger exit에서 nullptr를 허용하는가

처음에는 `Exit` 이벤트에서도 항상 상대 collider가 살아 있을 것처럼 생각하기 쉽지만, 실제로는 아래 상황이 자주 생깁니다.

- 상대 오브젝트가 삭제 예약됨
- active / trigger 상태 전환으로 contact가 강제 종료됨
- collider release 중 pair가 먼저 정리됨

이때 억지로 항상 상대 포인터를 유지하려고 하면 dangling pointer 위험이 커집니다.

그래서 현재는 아래 정책을 명시했습니다.

- `OnCollisionExit(nullptr)`
- `OnTriggerExit(nullptr)`

이 정책은 “이벤트 인자가 항상 유효 객체여야 한다”는 가정은 깨지만, 삭제/종료 경로를 안정적으로 처리하는 데 훨씬 유리했습니다.

## 10. 왜 테스트 씬과 샘플 씬을 분리해서 문서화했는가

프로젝트가 커지면서 `Sample.json`, `Physics2DTestScene.json`, `ActivePhysicsToggleTestScene.json`, `Physics2DInteractionTestScene.json`의 성격이 섞여 보이기 시작했습니다.

문서화 전에는 이런 문제가 있었습니다.

- 샘플 씬인지 검증 씬인지 혼란스러움
- “미사용 테스트 코드”처럼 보이지만 실제로는 SceneData에 사용 중인 컴포넌트가 있음
- 삭제 후보 판단이 자꾸 틀어짐

실제로 `CollisionVisualTestComponent`도 초기에는 제거 후보처럼 보였지만, 확인해 보니 `SceneData/Sample.json`에서 실제 사용 중이어서 샘플 유지로 다시 분류했습니다.

그래서 현재는 아래처럼 역할을 분리해 기록하고 있습니다.

- `Sample.json`
  - 예시 씬
- `Physics2DTestScene.json`
  - Physics2D 검증 씬
- `Physics2DInteractionTestScene.json`
  - interaction 검증 씬
- `ActivePhysicsToggleTestScene.json`
  - active/trigger/UI 연동 검증 씬

## 11. 왜 deprecated CollisionManager를 바로 삭제하지 않았는가

`CollisionManager.h/.cpp`는 더 이상 활성 충돌 시스템이 아니고, 실제 흐름은 Box2D `ContactListener` 기반입니다.

그럼에도 바로 파일을 삭제하지 않은 이유는 다음과 같습니다.

- 빌드 테스트 없이 바로 지우는 건 위험할 수 있었기 때문
- 옛 include나 프로젝트 항목 잔재를 먼저 확인할 필요가 있었기 때문
- 지금 단계 목표가 “구조 정리”이지 “공격적 삭제”는 아니었기 때문

그래서 이번에는 아래까지만 진행했습니다.

- 활성 사용처 확인
- `vcxproj` / `filters` 항목 제거
- 디스크 파일은 deprecated 후보로 유지

즉, “활성 코드에서 분리”를 먼저 끝내고, 실제 삭제는 후속 단계로 넘긴 상태입니다.

## 12. 왜 MainFrame / WindowFrame 경계를 먼저 분석만 했는가

`MainFrame`, `WindowFrame`은 현재 프로젝트에서 가장 많은 공통 흐름이 몰려 있는 축입니다.

- loop
- device
- physics
- collision dispatch
- resize/reset
- editor playback
- scene open 정책
- ImGui bootstrap

이런 파일은 성급히 뜯으면 한 번에 많은 경로가 깨질 수 있어서, 먼저 “어디까지가 Runtime이고 어디부터가 Editor인지”를 분석하는 단계가 필요했습니다.

현재 분석 결과는 대체로 아래 방향입니다.

- `MainFrame`
  - runtime core + editor playback / ImGui bootstrap 혼합
- `WindowFrame`
  - Win32 window core + scene open/app policy 혼합

그래서 지금은 수정 전에 경계선부터 기록하고, 후속 리팩터링은 책임 단위별로 자르는 방식이 더 안전하다고 보고 있습니다.

## 13. 최근 정리에서 얻은 운영 원칙

최근 문서 정리와 구조 점검을 거치면서 사실상 아래 원칙이 굳어졌습니다.

- 테스트/샘플 코드도 실제 씬에서 쓰고 있으면 함부로 제거하지 않는다.
- SceneData는 원본과 BuildOutput을 구분해서 본다.
- rename이 끝나도 legacy alias는 잔존 데이터 확인 전까지 바로 지우지 않는다.
- editor 기능이 runtime에 섞여 있더라도, 당장은 “기능 유지 -> 경계 문서화 -> 후속 분리” 순서를 택한다.
- 코드 정리 전에는 먼저 사용처, 프로젝트 항목, SceneData 사용 여부를 좁은 범위에서 확인한다.
