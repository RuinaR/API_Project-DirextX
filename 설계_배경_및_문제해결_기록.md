# 설계 배경 및 문제 해결 기록

이 문서는 "왜 이런 구조를 선택했는지"와
"작업하면서 어떤 문제를 만났고 어떻게 정리했는지"를 기록한 문서입니다.

최신 상태를 빠르게 보고 싶다면 `README.md`, `프로젝트_정책_및_가이드.md`를 먼저 보는 편이 좋습니다.

## 1. 왜 점진적으로 고치고 있는가

이 프로젝트는 처음부터 완전히 새 엔진 구조로 시작한 것이 아니라,
기존에 돌아가던 프레임워크를 바탕으로 기능을 붙여 오면서 점점 정리해 가는 방식으로 작업했습니다.

그래서 지금도 기준은 아래와 같습니다.

- 동작하던 기능을 최대한 유지할 것
- SceneData와 테스트 씬을 쉽게 깨지 않게 할 것
- 구조를 조금씩 분리하되, 한 번에 너무 크게 갈아엎지 않을 것

이 방향을 택한 이유는,
엔진 구조를 정리하는 과정 자체를 보여 주는 것도 포트폴리오에서 의미가 있다고 봤기 때문입니다.

## 2. 프로젝트 이름과 실행기 구조를 바꾼 이유

처음에는 실행기 이름이 아래처럼 되어 있었습니다.

- `BuildEdit`
- `BuildResultGame`

이 이름은 내부 작업 단계에서는 괜찮지만,
프로젝트 구조를 설명할 때 역할이 잘 보이지 않는 문제가 있었습니다.

그래서 아래처럼 정리했습니다.

- `BuildEdit` -> `EditorApp`
- `BuildResultGame` -> `GameApp`

이렇게 바꾼 뒤에는
"이 프로젝트가 editor 실행기인지, game 실행기인지"가 이름만 봐도 조금 더 잘 드러나게 됐습니다.

## 3. EngineFramework를 DLL로 바꾼 이유

처음에는 `EngineFramework`가 static library였습니다.
그런데 이 상태에서는 `MainFrame` 같은 싱글턴이 EXE와 DLL에 각각 따로 생기는 문제가 있었습니다.

실제로 겪었던 문제:

- EXE 쪽에서 `MainFrame::Create()`
- DLL 쪽에서 `MainFrame::GetInstance()->Update()`
- 서로 다른 인스턴스를 보게 되어 실행 직후 종료나 null 접근이 발생

이 문제를 줄이기 위해 `EngineFramework`를 `EngineFrameworkDll`로 바꾸고,
공통 엔진 쪽 상태를 DLL 쪽으로 모으는 방향으로 정리했습니다.

## 4. plugin 표면을 줄인 이유

처음 plugin 인터페이스에는 쓰이지 않거나 의미가 애매한 함수가 꽤 남아 있었습니다.

예:

- `AddObject`
- `RemoveObject`
- `Shutdown`
- `About`
- `Execute`
- `AllStart`
- `AllUpdate`

이 함수들은 실제 호출처가 없거나 no-op인 경우가 많아서,
plugin 역할을 이해하기 어렵게 만들었습니다.

그래서 현재는 의미가 남는 쪽으로 줄였습니다.

- `RegisterGameContent(...)`
- `GetName()`
- `ReleaseGameContent()`

이렇게 줄이면서
`KirbyGameDll`이 "게임별 등록/해제 역할을 맡는 DLL"이라는 방향이 더 분명해졌습니다.

## 5. AppLaunchDesc를 넣은 이유

예전에는 startup scene 결정, mode, fallback 정책이 여러 곳에 흩어져 있었습니다.

그래서 실행 정책을 하나로 묶기 위해 `AppLaunchDesc`를 도입했습니다.

주요 값:

- `renderType`
- `startupSceneName`
- `requireExistingSceneData`
- `createNewSceneIfMissing`
- `targetFps`

이 구조를 넣은 뒤,
EditorApp과 GameApp의 시작 정책을 같은 형식으로 설명하기 쉬워졌습니다.

## 6. MainFrame 생명주기를 App 쪽으로 옮기며 알게 된 점

한 번은 `MainFrame::Create / Update / Destroy`를 EXE 쪽으로 완전히 옮기려고 시도했습니다.
하지만 static lib 시절 구조의 영향 때문에 문제가 생겼습니다.

결론:

- 단순히 호출 위치만 옮기는 것으로는 충분하지 않았다.
- 엔진 공통 상태가 어느 모듈에 살아 있는지가 중요했다.

이 경험 때문에,
단순 리팩터링처럼 보여도 DLL 경계와 싱글턴 문제를 함께 봐야 한다는 점을 분명히 알게 됐습니다.

## 7. Runtime -> Editor 직접 의존을 끊은 이유

분리 작업을 하면서 특히 먼저 손본 곳은 아래 두 군데였습니다.

- `WindowFrame -> EditorSceneWorkflow`
- `RenderManager -> EditorRenderFacade`

이 둘은 runtime 쪽 코드가 editor 헤더를 직접 include하고 있었기 때문에,
나중에 `EngineEditor`를 분리하려고 하면 계속 발목을 잡는 구조였습니다.

그래서 callback 등록 방식으로 바꿨습니다.

- runtime은 callback만 알고 있음
- editor 쪽에서 실제 동작을 등록함

이건 나중에 `EngineEditor`를 더 분리할 때도 중요한 기준이 됩니다.

## 8. EngineEditor 분리를 바로 끝내지 못한 이유

`EngineEditor` 프로젝트는 만들었지만,
editor 코드를 한 번에 다 옮기지는 못했습니다.

이유:

- `EditorAssetField`를 runtime component가 직접 쓰는 경로가 있었음
- `EditorInspectorWindow`, `EditorSceneWorkflow`를 `KirbyGameDll` 쪽 테스트/사용자 코드가 직접 참조하는 부분이 있었음
- 일부 editor 코드를 옮기면 빌드는 되더라도 실제 include 경계가 꼬였음

그래서 현재는
"프로젝트 골격은 만들고, 직접 의존을 하나씩 끊어 가는 단계"로 보는 게 맞습니다.

## 9. ImGui와 DLL 경계 문제

최근에 실제로 겪었던 큰 문제 중 하나는,
user component의 inspector에서 `ImGui::Checkbox()` 호출 시 크래시가 난 일이었습니다.

원인:

- `EngineFrameworkDll`과 `KirbyGameDll`이 각자 ImGui current context를 따로 보게 됨
- inspector 호출 시 DLL 쪽 `GImGui == nullptr`

해결:

- `ImGuiContextBridge`를 추가
- inspector 시작 시 현재 모듈의 ImGui context를 엔진 쪽 context로 맞춰 주는 helper 사용

이 작업은
"DLL 경계를 넘는 UI 호출은 생각보다 조심해야 한다"는 점을 보여 준 사례였습니다.

## 10. 왜 Debug Log 창을 넣었는가

에디터 기능이 조금씩 늘어나면서,
Scene 저장/로드나 reference resolve, workflow 테스트 중에
화면 안에서 바로 볼 수 있는 간단한 로그가 필요해졌습니다.

그래서 1차로는 가볍게 아래 구조를 넣었습니다.

- `DebugLog` 저장소는 `EngineFrameworkDll`
- 표시 UI는 editor 쪽 창(`EditorDebugLogWindow`)
- GameApp에서는 창을 띄우지 않음

이건 디버깅 도구 자체보다,
"runtime 저장소 / editor 표시" 역할을 분리해서 생각하는 연습으로도 의미가 있었습니다.

## 11. ReferenceFieldRegistry를 넣은 이유

처음에는 inspector에서 참조 필드를 만들 때,
각 component가 `DrawInspector()` 안에서 직접 `DrawGameObjectReferenceField(...)` 같은 함수를 일일이 호출해야 했습니다.

이 방식은 간단하지만,
저장/로드나 공통 처리로 가기 어려운 문제가 있었습니다.

그래서 `ReferenceFieldRegistry`를 넣고,
component가 "이 필드는 GameObject 참조, 이 필드는 Component 참조"라고 등록만 하면,
inspector가 공통 방식으로 그리도록 정리했습니다.

그 뒤에 이어서:

- `componentId` 추가
- registry 기반 SceneData 저장/로드 추가
- hierarchy GameObject drop -> matching component 연결

까지 확장했습니다.

## 12. Component reference UX에서 배운 점

처음에는 component 자체를 drag해서 component reference 필드에 drop하는 방식도 시도했습니다.
하지만 inspector 안에서 component를 drag하는 UX는 실제로 써 보니 불안정했습니다.

문제:

- inspector selection/context가 흔들릴 수 있음
- drag 시작 지점이 어색함
- 사용자가 하이어라키 기반으로 생각하는 흐름과 잘 안 맞음

그래서 결국 더 자연스러운 방식으로 바꿨습니다.

- 하이어라키에서 `GameObject`를 drag & drop
- 대상 필드가 원하는 타입의 component를 그 오브젝트 안에서 찾아 연결
- 맞는 component가 없으면 아무 일도 하지 않음

이건 실제 사용성을 보고 수정한 좋은 사례라고 생각합니다.

## 13. SceneData 버전이 올라간 이유

최근 SceneData 버전은 아래처럼 올라왔습니다.

- v8
n  - component별 `componentId` 추가
- v9
  - registry 기반 `referenceFields` 추가

버전을 올린 이유는 단순히 "포맷이 바뀌었기 때문"만이 아니라,
이전 파일도 계속 읽을 수 있게 하기 위해서입니다.

즉,
최신 포맷은 더 많은 정보를 저장하지만,
구버전 SceneData도 가능한 한 계속 읽게 만드는 쪽을 유지하고 있습니다.

## 14. 현재 구조를 한 문장으로 정리하면

지금 KirbyEngine은
"기능을 만들어 보는 단계"를 넘어서,
"기능을 유지하면서 구조를 더 나은 방향으로 정리하는 단계"에 들어와 있습니다.

그래서 이 기록 문서는 완성된 설계서라기보다,
왜 이런 선택을 했는지 계속 남겨 두는 작업 노트에 가깝습니다.
