# 의존성 정책 정리

이 문서는 현재 프로젝트에서 어떤 프로젝트가 무엇을 참조하는지,
그리고 앞으로 어떤 방향으로 정리하려는지를 빠르게 보기 위한 문서입니다.

## 1. 현재 프로젝트 구성

현재 주요 프로젝트는 아래와 같습니다.

- `EditorApp`
- `GameApp`
- `EngineFrameworkDll`
- `EngineEditor`
- `KirbyGameDll`

## 2. 현재 의존 방향

### EditorApp

- `EngineFrameworkDll`
- `EngineEditor`
- 실행 중 `KirbyGameDll.dll` 로드

즉 editor 실행기는
엔진 공통 기능, editor 기능, 게임별 DLL을 함께 사용합니다.

### GameApp

- `EngineFrameworkDll`
- 실행 중 `KirbyGameDll.dll` 로드

즉 game 실행기는
editor 기능 없이 게임 실행에 필요한 쪽만 사용합니다.

### KirbyGameDll

- `EngineFrameworkDll`
- `EngineEditor`는 직접 참조하지 않는 방향을 유지합니다.

즉 게임별 DLL은
가능하면 runtime 쪽 공통 엔진 기능만 참조하는 구조를 목표로 둡니다.

### EngineEditor

- `EngineFrameworkDll`

즉 editor 기능은
공통 엔진 기능 위에 올라가는 별도 층으로 봅니다.

## 3. 왜 이런 방향을 유지하는가

이 구조를 두는 이유는 간단합니다.

- `EditorApp`은 editor 기능이 꼭 필요합니다.
- `GameApp`은 editor 기능이 없어야 더 단순합니다.
- `KirbyGameDll`은 게임별 코드에 집중해야 합니다.

즉
"게임 실행에 필요하지 않은 editor 코드가 game 쪽으로 섞이지 않게 하자"가 기본 원칙입니다.

## 4. 현재 아직 섞여 있는 지점

아직 완전히 분리된 상태는 아닙니다.

대표적으로 아래 파일들은 runtime/editor 책임이 일부 섞여 있습니다.

- `MainFrame`
- `WindowFrame`
- `RenderManager`

그래서 현재 프로젝트는
이름과 큰 방향은 먼저 정리했고,
세부 책임 분리는 계속 진행 중인 상태로 보는 편이 맞습니다.

## 5. 현재 기준으로 유지하는 규칙

### 좋은 방향

- `EditorApp` -> `EngineFrameworkDll`, `EngineEditor`
- `GameApp` -> `EngineFrameworkDll`
- `KirbyGameDll` -> `EngineFrameworkDll`
- `EngineEditor` -> `EngineFrameworkDll`

### 피하려는 방향

- `GameApp` -> `EngineEditor`
- `KirbyGameDll` -> `EngineEditor`
- runtime core가 editor 헤더를 직접 include하는 구조

## 6. 최근 정리한 부분

최근에는 아래 직접 의존을 줄였습니다.

- `WindowFrame -> EditorSceneWorkflow` 직접 호출 제거
- `RenderManager -> EditorRenderFacade` 직접 호출 제거

대신 callback 등록 방식으로 바꾸었습니다.
즉 runtime 쪽이 editor 코드를 직접 모르도록 정리한 셈입니다.

## 7. 왜 EngineFramework를 DLL로 바꾸었는가

예전 `EngineFramework`는 static library였습니다.
그런데 이 상태에서는 EXE와 DLL이 엔진 싱글턴을 따로 가지는 문제가 있었습니다.

예를 들어 아래처럼 보이는 상황이 생길 수 있었습니다.

- EXE에서 `MainFrame::Create()`
- DLL에서 `MainFrame::GetInstance()->Update()`

겉보기에는 같은 전역 객체를 쓰는 것 같지만,
실제로는 서로 다른 인스턴스를 볼 수 있었습니다.
이 때문에 실행 직후 종료되거나 null 접근이 생길 수 있었습니다.

그래서 공통 엔진 상태를 한쪽으로 모으기 위해
엔진 공통부를 `EngineFrameworkDll`로 정리했습니다.

## 8. EngineFrameworkDll은 어떻게 로드되는가

`EngineFrameworkDll`은 `KirbyGameDll`처럼 `PluginManager`가 직접 `LoadLibrary`로 읽는 DLL이 아닙니다.

현재 구조에서 `EngineFrameworkDll`은
`EditorApp`, `GameApp`, `KirbyGameDll`이 미리 링크해 두는 공통 엔진 DLL입니다.
즉 실행 파일이나 게임 DLL이 시작될 때,
운영체제가 import table을 보고 자동으로 함께 로드하는 방식으로 사용합니다.

반대로 `KirbyGameDll`은 실행 중에 `PluginManager`가 직접 읽는 DLL입니다.
즉 두 DLL의 성격은 아래처럼 다릅니다.

- `EngineFrameworkDll`: 프로그램이 돌아가기 위해 먼저 잡히는 공통 엔진 DLL
- `KirbyGameDll`: 나중에 플러그인처럼 읽어 오는 게임별 DLL

이 차이를 기준으로 보면,
`EngineFrameworkDll`은 엔진 본체에 가깝고,
`KirbyGameDll`은 게임 확장부에 가깝습니다.

## 9. 지금 단계에서의 정리

현재 의존성 정책은
"game 실행 경로에는 editor 의존을 최대한 줄이고,
엔진 공통 기능은 `EngineFrameworkDll`에 모으고,
게임별 확장은 `KirbyGameDll`로 분리한다"로 이해하면 됩니다.
